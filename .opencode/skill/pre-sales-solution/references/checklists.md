# 售前方案写作检查清单

## 通用红线（绝不能犯）

- [ ] 通篇堆产品参数，没有一句提到客户痛点
- [ ] 只画大饼，不说具体怎么做
- [ ] 用"显著提升""大幅降低"等模糊表述，不用具体数据
- [ ] 方案范围覆盖 10 个场景，每个都一笔带过
- [ ] 技术方案过度超前，客户 IT 基础根本接不住

---

## 第一段：需求痛点检查

### 基础项
- [ ] 引用了**具体政策文件名称或文号**
- [ ] 用一句话描述客户**当前业务状态**
- [ ] 识别了至少 3 个核心痛点（业务/技术/管理各 1 个）

### 数据检查
- [ ] 痛点 1 包含**量化数据**（金额/时间/数量/比例）
- [ ] 痛点 2 包含**量化数据**
- [ ] 痛点 3 包含**量化数据**
- [ ] 没有出现"显著""大幅""严重"等模糊词（除非后面跟具体数据）

### 深度检查
- [ ] 痛点之间有**逻辑层次**（不是简单罗列）
- [ ] **分析了痛点的根本原因**，不只是描述症状
- [ ] **建立了紧迫感**（不解决的后果 = 规避损失 > 追求收益）

### 常见问题
| ❌ 踩坑写法 | ✅ 正确写法 |
|-----------|-----------|
| "耕地保护形势严峻" | "2024 年度国土变更调查中发现非农化图斑 1,247 个，较去年同期增长 23%" |
| "传统监测效率低下" | "人工判读 2000 平方公里影像需 60 个工作日，而季度考核要求 30 天内完成" |
| "客户对信息化有需求" | "目前 80% 的业务流程还在纸质审批，平均办结时间 7.2 天" |

---

## 第二段：方案设计检查

### 场景聚焦
- [ ] 方案只聚焦**1-2 个核心场景**（不是大而全的全覆盖方案）
- [ ] 场景定义**用一句话说清**"谁 + 在什么情况下 + 要解决什么问题"
- [ ] 明确标注"本次方案**不覆盖**的场景范围"

### 技术匹配
- [ ] 技术路线与客户**现有技术基础**匹配（不要推 1000 万的方案给只有 50 万预算的客户）
- [ ] 每个技术选型都说明了**为什么选这个**，不是简单列功能
- [ ] 有方案架构图（可以是文字描述建议绘图）

### 竞品对比
- [ ] 列出了与**传统方案/竞品**的对比维度（效率/成本/准确率/落地难度）
- [ ] 对比维度用**数据**支撑，不是主观评价
- [ ] 突出 1-2 个**差异化亮点**，不求面面俱到

### FABE表达检查
- [ ] 技术优势是否转化为**客户利益**（Feature→Advantage→Benefit）
- [ ] 每个核心优势都有**Evidence证据**（案例、数据、引用）

### 常见问题
| ❌ 踩坑写法 | ✅ 正确写法 |
|-----------|-----------|
| "采用业界领先的 AI 大模型技术" | "基于遥感大模型语义分割技术，实现耕地地块自动识别，无需人工标注" |
| "平台功能丰富，支持 XXX" | "针对耕地监测场景，提供地块识别、作物分类、非农要素检测三个核心功能" |
| "相比传统方案效率更高" | "传统人工方案需 60 个工作日，本方案 16 小时内完成，效率提升 90 倍" |

---

## 第三段：实施保障检查

### 阶段规划
- [ ] 实施阶段**不超过 4 个**（太多显得复杂，太少显得粗糙）
- [ ] 每个阶段有**明确交付物**（不能只说"完成系统部署"）
- [ ] 每个阶段有**可量化的验收标准**

### 团队与质量
- [ ] 项目经理有**相关资质**（如系统架构师/项目管理师/PMP）
- [ ] 技术负责人有**同类项目经验**（如有，写明项目和规模）
- [ ] 质量管控有**具体流程**（如三级检查制度、数据精度验证）

### 风险与售后
- [ ] 识别至少 2 个**核心风险类型**
- [ ] 每个风险有**具体应对措施**，不说"加强管理"之类的空话
- [ ] 售后运维明确**服务频率**和**响应时间**

### 常见问题
| ❌ 踩坑写法 | ✅ 正确写法 |
|-----------|-----------|
| "我们将组建专业团队保障项目" | "配置 PMP 认证项目经理 1 名、5 年遥感项目经验的算法工程师 2 名" |
| "严格把控质量" | "执行数据自检→交叉检查→终审验收的三级质检，数据精度不低于 95%" |
| "提供完善的售后服务" | "提供 12 个月免费运维，7×24 小时响应，重大问题 4 小时内到场处理" |

---

## 方案整体完整性检查

- [ ] 三段之间**逻辑递进**（痛点→方案→保障，不是各说各的）
- [ ] 客户三问都有**明确回答**（懂不懂我？行不行？稳不稳？）
- [ ] 没有出现**前后矛盾**的描述
- [ ] 所有数据、政策引用**来源可查**
- [ ] 方案长度与项目规模**匹配**（不能小项目写 80 页，大项目写 10 页）
- [ ] 专业术语**有解释**（不要假设客户懂技术细节）

### 进阶：格局高度检查
- [ ] **时间维度**：是否有"从行业趋势看，过去→现在→未来"的拔高表述
- [ ] **空间维度**：是否有"从本部门到全局，再到生态/平台化"的扩展视野
- [ ] **价值维度**：是否有超越"解决具体问题"的拔高（如"不仅解决您问题，还能评先进、当标杆"）

## 汇报临场技巧检查表

### 会前准备
- [ ] 是否准备了**领导必问6类问题**的三层答案（数据来源/竞品对比/价格成本/风险/周期/案例）
- [ ] 是否准备了应对不同角色的**分层介绍话术**（老板/中层/技术）
- [ ] 是否准备了**无稿开口逻辑树**（一句话主题→3个论点→每个论点配1案例1数据）

### 会中沟通顺序
- [ ] 是否按照**现状摸底→痛点诊断→目标澄清→边界确认→方案验证**的五阶顺序沟通（尤其面对技术负责人）
- [ ] 是否避免一坐下就讲公司介绍，先听客户说
- [ ] 是否让客户自己说出痛点，而不是替他定义痛点

### 临场救急
- [ ] 被问住时是否能用**三层答案法**稳住（先直接答→再解释→最后升华价值）
- [ ] 是否准备了**痛点数据化**的话术来强化紧迫感（"不是效率低，是每个月损失200万订单"）

---

## 快速自检口诀

> **痛点要有数，方案要聚焦，保障要具体，格局要拔高。**
>
> - 痛点段 → 用数据说话，用量化建立紧迫感
> - 方案段 → 聚焦场景，不炫技，用FABE转客户利益
> - 保障段 → 明确交付，不画饼，可量化验收
> - 汇报段 → 分层沟通，三层答案，无稿开口
