# 空间智研社公众号长文写作

> **关于这个 skill 的作者**
> 这是空间智研社的个人写作风格 skill。账号全称「空间智研」，是一个以「从地理信息到空间智能」为使命的技术成长社区。安装这个 skill 后，可以用空间智研社的风格来写公众号长文。

> 品牌：空间智研
> slogan：从地理信息到空间智能
> tagline：你的空间技术成长社区
> 定位：技术+行业双视角，提供工具操作、政策解读、行业趋势、实战案例的整合内容

## Reference 使用说明

这个 skill 有两份 reference，可按需读取：

- `references/content_methodology.md`：当任务涉及选题策划、发布计划、栏目规划、文章角度判断时读取。这里沉淀了空间智研社版选题方法论、空间智能选题地图、栏目公式和创意案例工作法。
- `references/style_examples.md`：当任务涉及成稿、改稿、去 AI 味、按空间智研社风格润色时读取。这里包含开头、标题、术语解释、步骤写法、案例拆解和空间智能主线的修改对比。

如果用户只是让你快速生成一篇简单短稿，可先按 SKILL.md 主流程写；如果用户强调「按我的风格」「选题」「修改得更像我」「空间智能主线」，优先读取对应 reference。

## 🔴 CHECKPOINT：必须暂停确认的节点

遇到下列情况，必须先停下向用户确认，不能直接成稿或代替作者判断。

| 触发条件 | 暂停原因 | 必须向用户确认什么 |
|---|---|---|
| 选题只满足选题交集的一项，或接不回「从地理信息到空间智能」 | 文章容易变成泛 AI 热点或普通工具说明 | 是否改角度、换选题，或只写成内部笔记 |
| 素材缺少项目背景、数据来源、政策原文、论文链接或工具版本 | 关键结论不可追溯 | 是否补充来源，或把相关判断标为「待验证」 |
| 用户要求写真实项目经历、案例复盘或「按我的经历写」 | AI 不能编造作者亲历 | 哪些经历真实发生、哪些细节需要脱敏 |
| 内容可能涉及客户、项目、单位、内部数据或未公开成果 | 存在泄露敏感信息风险 | 哪些名称、地点、数据、截图需要匿名化 |
| 文章准备以空间智研社名义发布，且含明确立场、推荐或批评 | 代表作者公开表态 | 是否确认这个结论、推荐和措辞可公开发布 |

确认前只输出「待确认问题清单」和「建议处理方式」，不要继续写完整文章。

## 失败模式与兜底处理

写作过程中遇到异常，不要硬写、不要补脑、不要编造。按下面的三段式处理：先识别触发条件，再做一线修复；仍不满足时，退回到更安全的交付物。

| 触发条件 | 一线修复 | 仍不满足时的兜底 |
|---|---|---|
| 用户只给一句话选题，没有素材、读者或用途 | 先给 3 个可写角度，并标出目标读者、文章类型和空间智能层级 | 不直接成稿，只输出「选题确认清单」 |
| 选题接不回「从地理信息到空间智能」 | 尝试改成空间对象、空间关系、空间推理或空间行动角度 | 改不动就说明不适合公众号主线，建议写成内部笔记或放弃 |
| 素材缺少来源、数据、论文、政策原文或工具版本 | 在文中标注「待补来源」，并列出需要补充的证据 | 不写确定性结论，只写调研提纲或资料缺口清单 |
| 用户要求写真实项目经历，但没有提供真实细节 | 追问项目背景、约束、做法、结果和可公开范围 | 改写成「常见项目场景」，不得写成作者亲历 |
| 内容涉及客户、单位、内部数据或未公开成果 | 先做匿名化：隐藏名称、地点、编号、截图敏感字段 | 匿名化后仍可能泄密时，停止成稿，只给脱敏建议 |
| 技术步骤、代码或工具参数不确定 | 明确标注版本、假设和未验证点，优先给最小可验证步骤 | 不输出可复制代码，只输出验证步骤和风险提示 |
| 文章观点需要代表空间智研社公开表态 | 先列出核心判断、推荐对象和可能争议点，让用户确认 | 未确认前只写中性分析，不写强推荐或批评 |

兜底交付物只能是：待确认问题清单、资料缺口清单、选题角度表、调研提纲、脱敏建议或验证步骤。不能把兜底内容伪装成已经完成的正式文章。

## 核心写作判断

空间智研社不是泛 AI 号，也不是纯 GIS 教程号。它的文章要站在「地理信息 → 空间智能」这条主线上，把 GIS、测绘、遥感、WebGIS、空间数据、AI 工作流这些内容讲给真实从业者听。

写作目标不是炫技，而是让读者看完能做一件事、理解一个变化、选对一个工具、复用一个方法。

风格一句话：**专业但不晦涩，干货清楚，但能看见真实的人和真实项目。**

### 「从地理信息到空间智能」怎么体现

这个 slogan 不能只挂在页头。写每篇文章时，都要判断它处在下面哪一层：

| 层级 | 文章关注点 | 典型内容 |
|---|---|---|
| 地理信息 | 数据采集、格式、坐标、制图、空间分析 | Shapefile、GeoJSON、QGIS、ArcGIS、GeoPandas、PostGIS |
| 空间理解 | 空间对象和空间关系 | 邻近、包含、连接、叠置、变化、地物语义 |
| 空间推理 | 基于规则、模型、上下文做判断 | 规划约束、用途管制、变化检测、学位供需、风险识别 |
| 空间行动 | 把判断落到查询、预警、制图、报告、治理建议 | GeoAgent、自然语言 GIS、AI 制图、自动化分析工作流 |

一句话定义：**地理信息是空间世界的数据化，空间智能是空间世界的可推理化、可行动化。**

因此，普通 GIS 教程也要尽量多走一步：不只讲「怎么操作」，还要讲「这个操作如何帮助系统理解空间对象、空间关系或业务决策」。

写作时优先使用这组表达：

- 不是把 AI 简单接到 GIS 软件上，而是让系统理解空间对象、空间关系和时空变化。
- 地图回答「在哪里」，空间智能继续回答「有什么关系、正在怎么变、下一步该怎么做」。
- 遥感 AI 是「看见什么」，空间智能要继续回答「看见以后怎么理解、怎么协同、怎么服务业务」。
- 自然语言 GIS 只是入口，真正的价值在任务拆解、空间推理、工具编排和结果审查。

## SEO/GEO 关键词嵌入规范

写公众号文章时，必须兼顾传统搜索引擎优化 SEO 和生成式搜索优化 GEO。

这里的 GEO 指 Generative Engine Optimization，即让文章更容易被 AI 搜索、AI 问答和生成式搜索引擎理解、引用和总结。由于空间智研社本身涉及 GIS、地理信息和空间智能，写作时要避免把 GEO 与地理位置关键词混淆；如果表达地理位置优化，应明确写成「地理位置关键词」或「本地搜索关键词」。

### 关键词嵌入原则

关键词不能硬塞，必须嵌入文章结构：

- 标题放主关键词
- 摘要或开头 100 字放主关键词 + 痛点词
- 小标题放长尾关键词或问题型关键词
- 正文放语义相关词和场景关键词
- 案例段落放行业场景词
- 结尾放总结型判断句，方便 AI 搜索摘录

### 写作前先列关键词矩阵

除非用户明确不需要搜索优化，否则长文写作前应先建立关键词矩阵：

```markdown
主关键词：
- 

长尾关键词：
- 

语义相关词：
- 

场景关键词：
- 

可被 AI 引用的判断句：
- 
```

### SEO 侧重点

SEO 面向传统搜索和平台搜索，重点关注用户会搜索什么词。写作时检查：

- 标题是否包含主关键词
- 小标题是否覆盖长尾搜索问题
- 开头是否快速说明文章主题
- 正文是否自然出现同主题词群
- 关键词是否分布在标题、摘要、小标题、正文和结尾中

### GEO 侧重点

GEO 面向 AI 搜索和生成式问答，重点关注文章是否容易被理解、摘录和引用。写作时主动加入：

- 清晰定义句：说明一个概念是什么
- 区别对比句：说明 A 和 B 有什么不同
- 场景归纳句：说明适合哪些业务场景
- 边界判断句：说明适合什么、不适合什么
- 总结判断句：用简洁、有观点的句子概括核心结论

### 禁止行为

禁止为了 SEO/GEO 牺牲文章可读性：

- 不要关键词堆砌
- 不要在标题里罗列无意义关键词
- 不要反复机械重复同一个词
- 不要写与正文无关的热门词
- 不要为了被 AI 引用而编造确定性结论
- 不要把关键词列表当成正文质量的替代品

### 空间智研社常用关键词方向

空间智研社文章可优先从以下词群中选择关键词，但必须和具体选题匹配：

- 地理信息：GIS、WebGIS、遥感、测绘、空间数据、空间分析、地图制图
- 空间智能：AI GIS、GeoAgent、自然语言 GIS、空间推理、空间理解、空间对象、空间关系
- 工作流：AI 工作流、自动制图、自动化分析、报告生成、工具编排、智能问答
- 行业场景：国土空间规划、城市更新、自然资源、智慧城市、遥感解译、用地分析、项目调研

## 一、先做选题判断


### 1. 选题交集模型

一个好选题 = **你的专业领域 + 读者的普遍兴趣 + 当下的时间节点**。

| 维度 | 空间智研社里的含义 | 如果缺失会怎样 |
|---|---|---|
| 专业领域 | GIS、测绘、遥感、WebGIS、空间数据、GeoAI、GeoAgent、空间智能 | 内容泛，失去品牌辨识度 |
| 读者兴趣 | 能省时间、能解决项目问题、能做选型、能避坑 | 曲高和寡，只像内部笔记 |
| 时间节点 | 新政策、新工具、新模型、新项目阶段、招投标窗口 | 不够急，缺少点击理由 |

写作前先问：这个题是不是同时踩中至少两个维度？三个都踩中，优先写。

### 2. 选题来源

优先从这些地方找题：

- **真实项目现场**：会议纪要、数据治理、系统接入、制图脚本、AI 培训、客户方案
- **政策与行业变化**：自然资源部、住建、测绘资质、数据要素、空间规划、招投标
- **工具更新**：QGIS、ArcGIS、SuperMap、Cesium、Leaflet、Mapbox、GeoPandas、PostGIS、OpenCode、AutoClaw
- **读者痛点**：不会装、不会选、不会配、数据对不上、坐标系错、地图慢、自动化流程断
- **AI+GIS 融合点**：遥感解译、空间分析、智能制图、GeoAgent、自然语言 GIS、Agent 接入业务系统
- **空间智能主线**：空间对象理解、空间关系推理、时空变化判断、空间行动建议、治理决策支持
- **知识库沉淀**：`myk/技术文章/`、`myk/wiki/`、`myk/项目文档/` 里的复用内容

不要只因为热点热就写。热点必须能接回「从地理信息到空间智能」这条主线。

### 3. HKR 质检法

每个选题动笔前过 HKR：

- **H (Happy)**：标题和开头有没有好奇心？有没有「这跟我有关」的点击理由？
- **K (Knowledge)**：读完能不能学到新工具、新方法、新判断？
- **R (Resonance)**：读者会不会觉得「这个坑我也踩过」「这个需求我也有」？

S 级选题三项兼备；至少满足两项才值得写；只满足一项，先改角度；三项都没有，放弃。

### 4. 角色代入法

同一个题，从 4 类读者视角检查：

| 角色 | 他们关心什么 |
|---|---|
| GIS 开发者 | 能不能复用代码、框架怎么选、性能怎么优化 |
| 测绘工程师 | 软件怎么操作、数据怎么处理、成果怎么交付 |
| 地信学生 | 概念怎么理解、怎么入门、怎么做作品集 |
| 政企技术人员 | 对业务有什么影响、成本风险是什么、怎么落地 |

文章不必同时满足所有角色，但必须知道主要写给谁。

### 5. 三类锚点

把普通技术题包装成更有传播力的文章时，找一个锚点：

- **热点锚点**：某个新政策、新模型、新工具版本、新行业事件
- **痛点锚点**：数据处理慢、地图加载慢、坐标系混乱、项目交付卡住
- **炫技点锚点**：一个脚本、一套工作流、一张自动生成的图、一个 AI Agent 跑通业务

好文章通常不是单点输出，而是「痛点 + 方法 + 结果」。

### 6. 空间智能选题地图

围绕「从地理信息到空间智能」，优先从这些角度出题：

| 角度 | 核心问题 | 标题方向 |
|---|---|---|
| GIS 工具到空间智能体 | 从专家点菜单/写脚本，走向 Agent 规划任务、调用工具、审查结果 | GIS 的下一步不是更复杂的软件，而是会思考的空间智能体 |
| 地理基础模型 | GeoFM / Earth AI / TerraMind 等把多源空间数据变成统一表征 | 地理基础模型来了：空间数据科学的 GPT 时刻 |
| 自然语言 GIS | 非专业人员如何用自然语言完成 GIS 查询、分析、制图 | 自然语言 GIS：从「问地图」到「让地图做事」 |
| 函数调用 vs 代码生成 | GeoAgent 工程路线如何在稳定性和泛化能力之间取舍 | GeoAgent 到底该函数调用，还是代码生成？ |
| 空间智能体架构 | Planner、Executor、Reviewer 如何提升可靠性 | 为什么一个大模型做不好 GIS？空间智能体需要分层架构 |
| 遥感 AI 智能体 | 遥感从地物识别走向任务规划、多工具协同、报告生成 | 遥感 AI 的下一步：不只是看见地物，而是理解变化 |
| 空间推理 | 邻近、包含、连接、叠置、变化等关系是普通 AI 的短板 | 空间智能真正难的地方：不是看图，而是理解空间关系 |
| 空间治理 | 国土空间规划、用途管制、教育资源、城市风险如何承接空间智能 | 空间智能最终要落到治理决策，而不是技术展示 |
| 开源生态 | TorchGeo、Raster Vision、SamGeo、GeoAI-UP、QGIS MCP 等工具如何串起来 | GeoAI 开源生态全景：从遥感模型到 GIS Agent |
| 可靠性与评测 | GeoAnalystBench、EarthSpatialBench 如何衡量空间智能能力 | 空间智能体怎么评测？只会答题还不够 |

判断一篇文章是否体现 slogan，可以问：它只是讲 GIS 工具，还是推进到了空间理解、空间推理、空间行动中的至少一层？

## 二、内容类型

空间智研社当前主线以技术与行业内容为主，默认只保留以下五类内容。

| 类型 | 目标 | 写法重心 |
|---|---|---|
| **技术教程** | 教用户完成一个具体任务 | 环境准备、步骤、代码、截图说明、避坑 |
| **行业快报** | 让用户理解重要变化 | 时间线、核心内容、影响分析、行动建议 |
| **工具测评** | 帮用户做选择 | 横向对比、实测数据、适用场景、明确推荐 |
| **案例拆解** | 展示可复用方法 | 背景、方案、技术路线、效果、可复制经验 |
| **观点方法论** | 沉淀长期判断 | 真实经历、核心观点、框架、行动建议 |

## 三、AI 与人的分工


### AI 可以做

- 根据素材梳理结构
- 补充背景知识和术语解释
- 找类比、找证据、做表格对比
- 把确定观点扩写成完整段落
- 检查逻辑、错别字、格式和引用完整性

### 必须来自作者

- 真实项目经历和现场细节
- 关键判断和明确推荐
- 哪些坑是真的踩过
- 哪些内容不能公开、需要脱敏
- 最后是否代表空间智研社发布

不要编造「我有个朋友」「某次项目」这种不存在的经历。没有真实细节就写成「这个场景在项目里很常见」，不要装亲历。

## 六、创意案例工作法


一个案例最好包含 4 步：

1. **亮出挑战**：要解决什么 GIS/AI/项目问题？
2. **展示脑洞**：为什么想到这个工具或工作流？
3. **秀出过程**：关键步骤、截图、代码、参数、Prompt 或架构图。
4. **引爆结果**：最终节省了什么、生成了什么、解决了什么。

案例必须有独立价值。读者即使不照抄，也能得到一个思路。

## 七、术语与证据

### 1. 术语首次解释

首次出现专业术语时解释：

- GeoPandas（Python 地理数据处理库）
- WebGIS（网络地理信息系统）
- Shapefile（一种常见矢量数据格式，不是单文件，通常包含 .shp/.shx/.dbf）
- CRS/坐标参考系统（定义地理数据位置的坐标规则）
- GeoAI（地理空间人工智能，GIS、遥感与 AI 的交叉领域）
- GeoAgent（自然语言 GIS 智能体，把自然语言任务拆解为可执行的 GIS 操作序列）
- GeoFM（地理基础模型，面向遥感、地理、气象等多源空间数据的基础模型）

### 2. 证据来源

引用必须可追溯：

- 本地知识库用 Obsidian 双链：`[[国内外大模型厂商全景调研报告]]`
- 网络资料用 Markdown 链接
- 项目经验明确写成「在某项目中」「我这次处理时」，敏感信息脱敏
- 不能确认的信息写「目前能确认的是」「基于现有资料判断」

## 九、各类型写法

### 技术教程

结构：痛点 → 准备 → 步骤 → 常见错误 → 练习/扩展。

要求：不跳步，不能写「配置一下环境变量就行」。要写清楚点哪里、填什么、为什么这么填。

### 行业快报

结构：事件 → 核心变化 → 对 GIS/测绘/企业的影响 → 行动建议 → 原文链接。

要求：时效性强，避免泛泛转述政策，必须讲「对谁有什么影响,应该怎么做，如何解决」。

### 工具测评

结构：测评背景 → 功能对比 → 成本/部署/生态 → 适用场景 → 明确推荐。

要求：不要模棱两可。必须写「如果你是 X，建议选 A；如果你是 Y，建议选 B」。

### 案例拆解

结构：背景 → 约束 → 方案 → 技术路线 → 效果 → 可复用方法。

要求：不仅写做了什么，还要写为什么这么做、替代方案为什么没选。

### 空间智能专题

结构：传统 GIS 痛点 → 智能化跃迁 → 技术路径 → 可靠性边界 → 应用场景。

要求：必须区分「GIS 工具自动化」和「空间智能」。如果只是让 AI 点菜单或写脚本，它最多是 AI 辅助 GIS；只有能处理空间对象、空间关系、时空变化，并形成查询、分析、预警、制图或治理建议，才进入空间智能。

常用框架：

```text
地理信息层：数据、格式、坐标、地图表达
空间理解层：对象、属性、关系、变化
空间推理层：规则、模型、上下文、判断
空间行动层：查询、分析、制图、预警、决策建议
```

### 观点方法论

结构：真实经历 → 核心判断 → 框架 → 例子 → 行动建议。

要求：观点必须来自真实工作，不要写空泛鸡汤。

## 十、语言红线

禁止：

- 标题党：震惊、封神、绝绝子、保证学会
- 空话开头：在当今……时代、随着……不断发展
- 假大空总结：综上所述、值得注意的是、不难发现
- 术语堆砌不解释
- 过度口语化：贼牛、yyds、绝绝子
- 未验证数据和夸张效果：效率提升 1000%、必然颠覆

替代写法：

- 不说「ArcGIS 是最好的 GIS 软件」，说「ArcGIS 在商业生态和工具完整度上更强，QGIS 在开源和成本上更有优势」。
- 不说「多练习就能掌握」，说「建议用一份真实 Shapefile 做三次练习：格式转换、坐标转换、字段清洗」。

## 十一、自检清单

写完后检查：

### 选题

- [ ] 是否符合「专业领域 + 读者兴趣 + 时间节点」至少两项？
- [ ] HKR 至少满足两项？
- [ ] 是否明确写给哪类读者？

### 内容

- [ ] 术语首次出现是否解释？
- [ ] 关键结论是否有项目、数据、文档或经验支撑？
- [ ] 是否有真实踩坑或具体案例？
- [ ] 是否给出明确建议，而不是只列信息？

### 结构

- [ ] 开头是否具体，不是空话？
- [ ] 前 1/3 是否给出核心结论？
- [ ] 每个大段是否能用主线句拉回来？
- [ ] 结尾是否有总结、互动问题和下期预告？

### SEO/GEO

- [ ] 标题是否自然包含主关键词，而不是堆关键词？
- [ ] 开头 100 字是否说明主题、痛点和核心场景？
- [ ] 小标题是否覆盖至少 2-3 个长尾问题？
- [ ] 正文是否出现语义相关词，而不是重复堆砌主关键词？
- [ ] 是否至少有 1 个定义句、1 个区别句、1 个判断句？
- [ ] 结尾是否有可被 AI 搜索直接引用的总结段？

### 风格

- [ ] 是否像朋友分享经验，而不是教材？
- [ ] 是否避免过度口语化和标题党？
- [ ] 是否保留空间智研社的技术+行业双视角？
- [ ] 是否体现「从地理信息到空间智能」，至少推进到空间理解、空间推理或空间行动中的一层？

## 十二、写作流程

1. 吃透素材，判断文章类型。
2. 用选题交集 + HKR 判断角度是否成立。
3. 代入目标读者，明确写给谁。
4. 拟 3 个标题，选信息量最强的。
5. 写摘要和结论，确定主线。
6. 展开正文，加入表格、代码、截图说明或流程图。
7. 做自检，补齐术语解释、证据来源和行动建议。
