# 会议纪要技能

把会议原始记录整理成**可追踪、可检查、可执行**的纪要。

## 核心原则

- **可追踪**：纪要里的结论和待办，必须能回到原始会议内容，不能凭空补
- **可检查**：读完以后要能看出有没有遗漏关键议题、负责人、时间节点和风险点
- **可执行**：会议不能写完就结束，最后要落到下一步行动

## 功能模块

本技能包含三个模块，按顺序执行：

| 模块 | 功能 | 输入 | 输出 |
|------|------|------|------|
| **A. 音频转录** | 音频 → 文本 | WAV/MP3/FLAC | transcript.json |
| **B. 纪要生成** | 文本 → 结构化纪要 | transcript.json | 会议纪要.md |
| **C. Claude 润色** | 纪要 → 专业纪要 | 会议纪要.md | 最终纪要.md |

**执行顺序**：A → B → C（可跳过 A，直接从 B 开始处理已有文本）

## 模块 A：音频转录

### 适用场景

当用户提供以下材料时，使用本模块：
- 会议录音文件（WAV/MP3/FLAC/M4A）
- 实时麦克风语音
- 需要将音频转为文本

### 转录命令

```bash
# 离线转录（推荐）
python .opencode/skills/meeting-minutes/scripts/transcribe.py -i "会议录音.wav" -o transcript.json

# 实时转录
python .opencode/skills/meeting-minutes/scripts/transcribe.py --realtime -o live.json
```

### 模型选择

| 场景 | 推荐模型 | 说明 |
|------|----------|------|
| 通用会议 | nano | 31种语言，带时间戳 |
| 情感分析 | sensevoice | 检测情绪和音频事件 |
| 实时转录 | streaming | 流式识别，低延迟 |
| 中文优化 | paraformer | 中文专精，准确率高 |

## 模块 B：纪要生成

### 适用场景

当用户提供以下材料时，使用本模块：
- 会议录音转写文本
- 会议手写记录
- 聊天记录式会议内容
- 要求整理会议纪要、提取会议待办、总结会议结论

### 输入要求

用户最好提供：
- 会议主题
- 会议日期
- 参会人员
- 原始会议记录
- 会议目标
- 输出用途：内部同步 / 领导汇报 / 项目跟进 / 对外纪要
- 特殊要求：是否脱敏、是否保留争议点、是否需要待办表格

## 信息不足时的处理

- 缺少会议主题、日期、参会人员时，**不编造**，标注"待补充"
- 缺少原始会议记录时，**先追问**，不直接生成纪要
- 负责人、截止时间、决策结果不明确时，标注"待确认"
- 涉及客户、项目编号、内部数据时，提醒用户确认是否需要脱敏

## 处理流程

1. 判断会议类型和输出用途
2. 检查输入是否完整，列出缺失信息
3. 从原始记录中提取主要议题
4. **区分事实、讨论、结论、待办和待确认问题**
5. 只把有明确动作的信息整理为待办事项
6. 对责任人、截止时间不明确的待办标注"待确认"
7. 按固定格式输出会议纪要
8. 输出前做质量检查

### 关键：区分四类信息

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| 事实 | "上个月营收 500 万" | 直接记录 |
| 讨论 | "这个方案下周再看" | 记录为讨论，不写成决策 |
| 结论 | "决定采用 A 方案" | 记录为结论 |
| 待办 | "张三下周三前提交方案" | 记录为待办（必须有责任人+动作+时间） |

**注意**："这个模块可能要调整一下"**不是待办**。"张三下周三前把这个模块的调整方案补一下"**才是待办**。

## 输出格式

### 会议基本信息

- 主题：
- 时间：
- 参会人员：
- 输出用途：

### 会议摘要

用 3-5 句话概括本次会议讨论内容和主要结果。

### 关键结论

- 结论 1：
- 结论 2：

### 议题记录

按议题分块，每个议题包含：
- **议题名称**
- **讨论要点**
- **当前结论**
- **相关风险**

### 章节索引（带时间戳）

基于转录时间戳生成章节索引，方便回听定位：

```markdown
## 章节

00:01:12 议题一：xxx
00:03:37 议题二：xxx
00:06:56 议题三：xxx
```

**生成规则**：
- 每个议题/话题切换点生成一个章节
- 标题为一句话概括该段内容
- 时间戳来自转录结果

### AI洞察（增值分析）

在纪要末尾添加分析模块，提供决策参考：

```markdown
## AI洞察

- **会议风格**：本次会议结构清晰/重点突出/节奏紧凑...
- **风险提示**：多项活动存在名额限制和时间重叠风险...
- **行动建议**：建议家长尽快决策是否参与...
- **策略价值**：国家级证书构成差异化优势...
```

**分析维度**：
1. 会议的整体风格和策略
2. 活动之间的关联和冲突
3. 需要优先决策的事项
4. 对参会人的价值点

### 待办事项（细化版）

把"大事项"拆成"可执行动作"，每项必须有：
- **动作**：具体要做什么
- **负责人**：谁来做
- **截止时间**：什么时候完成
- **标准**：做到什么程度

```markdown
## 待办

- [ ] 准备新世纪演出节目《习语》舞蹈，时长约三分半钟
- [ ] 统一新世纪演出服装，学生需穿粉色纱裙和黑色体操服
- [ ] 确认班级学生是否参加新世纪演出，不参加者需私信告知
- [ ] 组织暑期集训班报名，每班限18人，有意者私信报名
- [ ] 安排端午节放假事宜，6月19日周五当天放假不上课
```

**细化原则**：
- 把"安排演出"拆成"确认节目+准备服装+通知家长"
- 把"报名集训"拆成"发布信息+收集报名+确认名单"
- 每个动作都是"可执行"的，不需要再想"怎么做"

### 信息校验

对关键数字进行交叉验证，标注可能的转录错误：

```markdown
## 信息校验

| 信息点 | 转录内容 | 校验结果 | 备注 |
|--------|----------|----------|------|
| 北京比赛费用 | 520元 | 待确认 | 转录可能有误 |
| 考级时间 | 8月11日 | ✅ 一致 | |
```

**校验规则**：
- 费用、日期、人数等关键数字必须校验
- 如果同一信息在不同位置出现不一致，标注"待确认"
- 如果转录明显错误（如"五百二十"被识别为"九百二十"），标注原始内容

### 待确认问题

- 问题 1：
- 问题 2：

### 风险提醒

- 风险 1：

## 约束规则（红线）

1. **不编造会议中没有出现的结论**
2. **不把讨论意见写成最终决策**
3. **不把不确定责任人写成确定责任人**
4. **不把"建议""可能""后续再看"改写成已确认事项**
5. 涉及客户、项目编号、内部数据时，默认提醒脱敏
6. 对外版本必须提示用户确认可公开范围

### 信息分类标记

输出中必须区分：
1. **已确认事实**
2. **待验证信息**（疑似、可能、口头提到、未明确）
3. **推测或初步判断**
4. **需要人工确认的问题**

凡是材料中出现"疑似、可能、初步、待确认、未明确、口头提到"等表述，统一标注为"待验证"，**不得将待验证信息改写为确定事实**。

### 关键判断必须人工确认

涉及以下内容时，必须要求用户确认后再写入最终稿：
1. 会议决策和责任归属
2. 工具选型建议和强推荐
3. 对客户、项目、单位或方案的评价
4. 风险等级、合规判断、业务结论
5. 准备公开发布的文章、报告或对外材料

未确认前，只能写成"候选判断""待确认建议"或"需要人工复核"。

## 质量检查

输出前检查：
- [ ] 是否包含会议基本信息？
- [ ] 是否区分结论、待办和待确认问题？
- [ ] 每条待办是否有明确动作？
- [ ] 每条待办是否有责任人或标注"待确认"？
- [ ] 每条待办是否有时间节点或标注"待确认"？
- [ ] 是否出现了原文没有支撑的判断？
- [ ] 是否包含敏感信息提醒？
- [ ] 是否存在没有来源的事实？有则标注"待验证"或删除
- [ ] 是否存在材料没有支撑的结论？有则改为"待确认"
- [ ] 是否把推测写成事实？有则拆分为事实和推测

## Claude 润色流程

转录完成后，让 Claude 按以下步骤整理专业纪要：

### 步骤 1：读取转录结果

```bash
# 查看转录内容
cat transcript.json
```

### 步骤 2：提取关键信息

从转录文本中识别：
- **结论词**：决定、确认、同意、通过、否决
- **待办要素**：人名 + 动作 + 时间（张三 + 提交 + 下周三）
- **不确定表述**：可能、待定、再讨论、暂时、建议
- **风险词**：风险、问题、担心、延期、预算

### 步骤 3：按模板填充

使用 `references/minutes_template.md` 格式，按以下顺序填充：
1. 会议基本信息（主题、日期、参会人、用途）
2. 会议摘要（3-5 句话概括）
3. 关键结论（只列确定的结论）
4. 议题记录（每个议题的讨论要点 + 结论 + 风险）
5. 待办事项（必须有：事项 + 负责人 + 截止时间）
6. 待确认问题（不确定的事项）
7. 风险提醒（识别到的风险）

### 步骤 4：执行自检清单

逐项检查约束规则：
- [ ] 是否存在没有来源的事实？→ 标注"待验证"
- [ ] 是否存在材料没有支撑的结论？→ 改为"待确认"
- [ ] 是否把推测写成事实？→ 拆分事实和推测
- [ ] 是否保留了敏感信息？→ 脱敏处理

### 步骤 5：输出最终纪要

保存为 Markdown 文件，文件名格式：`{会议主题}_纪要_{日期}.md`

## 转录功能

### 离线转录

```bash
# 使用 Fun-ASR-Nano（推荐，31种语言）
python .opencode/skills/meeting-minutes/scripts/transcribe.py -i "会议录音.wav" -o transcript.json

# 使用 SenseVoiceSmall（情感+事件检测）
python .opencode/skills/meeting-minutes/scripts/transcribe.py -i "会议录音.wav" -m sensevoice -o transcript.json

# 指定语言（提升准确率）
python .opencode/skills/meeting-minutes/scripts/transcribe.py -i "会议录音.wav" -l zh -o transcript.json
```

### 实时转录

```bash
# 无限时长，Ctrl+C 停止
python .opencode/skills/meeting-minutes/scripts/transcribe.py --realtime -o live.json

# 指定时长（秒）
python .opencode/skills/meeting-minutes/scripts/transcribe.py --realtime -d 3600 -o meeting.json
```

### 生成纪要

```bash
python .opencode/skills/meeting-minutes/scripts/format_minutes.py -i transcript.json -o 会议纪要.md -t "项目周会" -p "张三" "李四"
```

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--input, -i` | 必填 | 输入音频文件路径 |
| `--output, -o` | stdout | 输出 JSON 文件路径 |
| `--model, -m` | nano | 模型：nano/sensevoice/streaming/paraformer |
| `--language, -l` | auto | 语言：auto/zh/en/ja/ko |
| `--device` | cuda | 设备：cuda/cpu |

### 模型选择

| 场景 | 推荐模型 | 说明 |
|------|----------|------|
| 通用会议 | nano | 31种语言，带时间戳 |
| 情感分析 | sensevoice | 检测情绪和音频事件 |
| 实时转录 | streaming | 流式识别，低延迟 |
| 中文优化 | paraformer | 中文专精，准确率高 |

## 技术异常处理

| 异常 | 处理方式 |
|------|----------|
| 音频格式不支持 | 提示用 ffmpeg 转换：`ffmpeg -i input.mp4 -ar 16000 -ac 1 output.wav` |
| GPU 显存不足 | 自动降级到 CPU 模式：添加 `--device cpu` 参数 |
| 模型下载失败 | 提示检查网络，或使用 ModelScope 镜像（已默认） |
| 实时转录无声音 | 检查麦克风权限，运行 `python -c "import pyaudio; p=pyaudio.PyAudio(); [print(i, p.get_device_info_by_index(i)['name']) for i in range(p.get_device_count())]"` 列出设备 |
| 转录结果为空 | 检查音频是否有语音内容，尝试其他模型 |
| JSON 解析错误 | 检查转录输出文件是否完整，重新转录 |
