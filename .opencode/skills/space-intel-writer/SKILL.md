# 空间智研社公众号长文写作

## Core rule

空间智研社的品牌线是「从地理信息到空间智能」。文章必须服务 GIS、GeoAI、WebGIS、遥感、空间数据、空间智能或 AI 工作流从业场景。

正式公众号文章 **必须包含 YAML frontmatter**，用于 Obsidian、知识库和后续检索。YAML 是归档元数据，不替代文末参考资料。

## Runtime 快速流程

1. 识别任务类型：选题、提纲、成稿、改稿、去 AI 味、标题、发布计划。
2. 判断是否命中暂停确认节点。命中时只输出「待确认问题清单」和「建议处理方式」。
3. 按任务路由读取 reference。只读当前任务需要的章节，不把 reference 当事实来源。
4. 写作前确认文章能接回「从地理信息到空间智能」。接不回时先改角度或建议作为内部笔记。
5. 正式成稿时输出 YAML frontmatter、正文和参考资料；标题/提纲/计划类任务只输出对应交付物。
6. 交付前执行最小自检：任务类型、品牌主线、事实来源、敏感信息、YAML、黑名单。

## 直接写作边界

| 情况 | 动作 |
|---|---|
| 用户提供明确主题、目标读者、用途和可公开素材 | 直接写提纲、标题或初稿 |
| 用户要求基于已有笔记、会议纪要、调研材料整理公众号文章 | 先按素材写作，不补编缺失事实；缺口用「待补来源」标出 |
| 用户只要标题、摘要、结构优化、去 AI 味或语言润色 | 直接处理，保留原事实和观点 |
| 选题脱离 GIS、GeoAI、WebGIS、遥感、空间数据、空间智能或 AI 工作流 | 先确认是否换角度，不直接成稿 |
| 涉及真实项目经历、客户单位、未公开数据、明确推荐或公开批评 | 先确认事实、脱敏范围和发布口径 |
| 素材缺少来源但要求输出确定结论 | 先追问来源；用户不补时只能输出调研提纲或资料缺口清单 |

## 正式文章输出契约

除非用户明确要求只要标题、提纲、摘要、发布计划或局部改写，正式公众号文章默认用 Markdown 输出，并且必须从 YAML frontmatter 开始。

```markdown
---
title: "文章标题"
author: "WY"
date: "YYYY-MM-DD"
category: "空间智研社"
type: "公众号文章"
brand: "从地理信息到空间智能"
topic: "GeoAI / WebGIS / 遥感 / AI工作流"
tags:
  - 空间智能
  - GIS
keywords:
  - 空间智能
  - 地理信息
summary: "一句话摘要，便于检索和列表展示"
source_type: "原创"
status: "draft"
---

# 标题

> 摘要：50-100 字，直接说明文章价值。

## 一、先说结论
## 二、背景或痛点
## 三、核心内容
## 四、避坑与建议
## 五、总结

---

## 参考资料
- 待补来源
```

### YAML 规则

- `title`、`author`、`date`、`category`、`type`、`brand`、`topic`、`tags`、`summary` 为必填。
- 日期使用 `YYYY-MM-DD`。
- 中文字符串建议加引号，避免冒号、斜杠等字符影响解析。
- `tags` 和 `keywords` 使用数组，不写成逗号字符串。
- `summary` 只写 1-2 句话，服务检索和列表预览。
- 正文引用仍放在 `## 参考资料`，YAML 不替代来源标注。

## SEO/GEO 使用规则

GEO 在这里指 Generative Engine Optimization，不是地理位置优化。写作时要让文章便于传统搜索、平台搜索和 AI 搜索理解，但不能牺牲可读性。

- 关键词矩阵默认作为内部规划，不主动输出。
- 用户要求 SEO/GEO 方案、AI 搜索优化、选题规划或发布计划时，才显式输出关键词矩阵。
- 成稿任务只在标题、摘要、小标题、正文和总结中自然体现关键词。
- 必须至少包含 1 个清晰定义句、1 个区别对比句、1 个边界判断句或总结判断句。
- 不要堆关键词、罗列无意义热门词或为了被 AI 引用而编造确定性结论。

## 🔴 CHECKPOINT：必须暂停确认的节点

遇到下列情况，必须先停下向用户确认，不能直接成稿或代替作者判断。

| 触发条件 | 暂停原因 | 必须向用户确认什么 |
|---|---|---|
| 选题只满足选题交集的一项，或接不回「从地理信息到空间智能」 | 文章容易变成泛 AI 热点或普通工具说明 | 是否改角度、换选题，或只写成内部笔记 |
| 素材缺少项目背景、数据来源、政策原文、论文链接或工具版本 | 关键结论不可追溯 | 是否补充来源，或把相关判断标为「待验证」 |
| 用户要求写真实项目经历、案例复盘或「按我的经历写」 | AI 不能编造作者亲历 | 哪些经历真实发生、哪些细节需要脱敏 |
| 内容可能涉及客户、项目、单位、内部数据或未公开成果 | 存在泄露敏感信息风险 | 哪些名称、地点、数据、截图需要匿名化 |
| 文章准备以空间智研社名义发布，且含明确立场、推荐或批评 | 代表作者公开表态 | 是否确认这个结论、推荐和措辞可公开发布 |
| 内容含政策、法规、招投标、资质或合规判断 | 时效和适用范围容易误导 | 发布日期、发布机构、原文链接、适用范围 |
| 内容含第三方截图、图片、Logo、论文图表或代码片段 | 可能有版权或授权风险 | 是否已授权，是否改成文字描述或自绘示意 |

确认前只输出「待确认问题清单」和「建议处理方式」，不要继续写完整文章。

## 失败模式与兜底处理

| 触发条件 | 一线修复 | 仍不满足时的兜底 |
|---|---|---|
| 用户只给一句话选题，没有素材、读者或用途 | 先给 3 个可写角度，并标出目标读者、文章类型和空间智能层级 | 不直接成稿，只输出选题确认清单 |
| 选题接不回「从地理信息到空间智能」 | 尝试改成空间对象、空间关系、空间推理或空间行动角度 | 改不动就说明不适合公众号主线，建议写成内部笔记或放弃 |
| 素材缺少来源、数据、论文、政策原文或工具版本 | 在文中标注「待补来源」，并列出需要补充的证据 | 不写确定性结论，只写调研提纲或资料缺口清单 |
| 用户要求写真实项目经历，但没有提供真实细节 | 追问项目背景、约束、做法、结果和可公开范围 | 改写成常见项目场景，不得写成作者亲历 |
| 内容涉及客户、单位、内部数据或未公开成果 | 先做匿名化：隐藏名称、地点、编号、截图敏感字段 | 匿名化后仍可能泄密时，停止成稿，只给脱敏建议 |
| 技术步骤、代码或工具参数不确定 | 明确标注版本、假设和未验证点，优先给最小可验证步骤 | 不输出可复制代码，只输出验证步骤和风险提示 |
| 工具测评需要推荐结论 | 未确认公开口径前只写条件性建议 | 确认版本、测试环境、样本限制后再写明确推荐 |

兜底交付物只能是：待确认问题清单、资料缺口清单、选题角度表、调研提纲、脱敏建议或验证步骤。不能把兜底内容伪装成正式文章。

## Do not do 黑名单

- 不要把泛 AI 热点硬写成空间智研社文章。
- 不要编造项目经历、客户反馈、实测数据、政策结论、论文观点或工具效果。
- 不要把 reference 当事实来源，它只提供方法和风格。
- 不要用「在当今时代」「随着不断发展」「综上所述」「值得注意的是」「不难发现」「赋能」「深度融合」「未来已来」「释放巨大潜能」这类套话。
- 不要标题党，不用震惊、封神、绝绝子、终极真相、保证学会、月入十万。
- 不要堆关键词、堆术语或写无关热门词。
- 不要跳过术语解释、版本说明、证据来源和适用边界。
- 不要在未确认前代表空间智研社做强推荐、强批评或公开承诺。
- 不要把内部资料、客户名称、项目地点、编号、截图敏感字段直接写进公开稿。

## 最小自检清单

交付前检查：

- [ ] 任务类型是否正确：标题、提纲、成稿、改稿、去 AI 味、发布计划没有混淆？
- [ ] 正式文章是否包含 YAML frontmatter，且字段服务检索？
- [ ] 是否体现「从地理信息到空间智能」？
- [ ] 是否明确目标读者和读者收益？
- [ ] 是否至少落到事实锚、场景锚或观点锚之一？
- [ ] 术语首次出现是否解释？
- [ ] 关键结论是否有项目、数据、文档或经验支撑？缺口是否标为「待补来源」？
- [ ] 是否触发客户、项目、政策、版权、公开表态等暂停确认节点？
- [ ] 标题、摘要、小标题和总结是否自然覆盖关键词，而不是堆砌？
- [ ] 是否至少有定义句、区别句、边界判断句或总结判断句？
- [ ] 是否避开 Do not do 黑名单？

## 参考资料写法

- 本地知识库使用 Obsidian 双链：`[[OpenCode入门指南]]`。
- 网络资料使用 Markdown 链接：`[标题](URL)`。
- 项目经验必须脱敏；不能确认的信息写「待补来源」或「待验证」。
- 如果没有任何来源，文末保留 `## 参考资料` 并写 `- 待补来源`。
