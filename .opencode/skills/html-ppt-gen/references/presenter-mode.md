# 演讲者模式指南 (Presenter Mode)

> 按 **S** 键弹出独立演讲者窗口，让演讲像呼吸一样自然。

## 架构总览

```
┌─────────────────┐       BroadcastChannel        ┌──────────────────────┐
│   主演示窗口      │ ◄──────────────────────────► │   演讲者窗口 (popup)  │
│   (观众端)       │                              │   (演讲者端)          │
│                 │                              │                      │
│  ┌───────────┐  │    postMessage               │  ┌────────┬────────┐ │
│  │ Slide N   │  │ ◄──────── flip signal        │  │当前页  │下一页  │ │
│  │ (iframe)  │  │                              │  │iframe  │iframe  │ │
│  └───────────┘  │                              │  └────────┴────────┘ │
│                 │                              │  ┌────────┬────────┐ │
│                 │                              │  │ 逐字稿  │ 计时器  │ │
│                 │                              │  │ panel  │ panel  │ │
│                 │                              │  └────────┴────────┘ │
└─────────────────┘                              └──────────────────────┘
```

四个磁吸卡片全部支持**拖拽位置**和**调整大小**，布局自动吸附到边缘。

## 核心技术机制

### BroadcastChannel 同步翻页

```js
// 演讲者窗口 → 主演示窗口
const channel = new BroadcastChannel('deck-sync');
channel.postMessage({ type: 'flip', page: 3 });
```

- 两个窗口通过 `BroadcastChannel('deck-sync')` 通信
- 无需服务端、无需 WebSocket、同一 tab 即可同步
- 关闭弹窗不影响主窗口展示

### iframe 预览机制

每个预览卡片是一个 `<iframe>`，加载同一份 deck HTML：

```
deck.html?preview=3   → 当前页（第 3 页）
deck.html?preview=4   → 下一页（第 4 页）
```

**runtime.js 检测 `?preview=N` 参数后：**
1. 只渲染第 N 页 slide
2. 隐藏所有 chrome（导航栏、页码、进度条）
3. 设置 `transform: scale()` 自适应 iframe 尺寸

### 翻页无白屏切换

翻页时**不重新加载 iframe**：

```js
// postMessage 通知 iframe 切换 .is-active class
iframe.contentWindow.postMessage({ type: 'go-to', page: 4 }, '*');
```

- iframe 内所有 slide 已预加载
- 仅切换 `.is-active` / `.is-hidden` class
- 零白屏、零闪烁、零重新渲染

## 快捷键一览

| 快捷键 | 作用 | 适用范围 |
|--------|------|---------|
| **S** | 打开演讲者窗口 | 主演示窗口 |
| **← →** | 翻页（同步到观众端） | 演讲者窗口 |
| **R** | 重置计时器 | 演讲者窗口 |
| **Esc** | 关闭弹窗 | 演讲者窗口 |
| **N** | 底部 notes 抽屉 | 主演示窗口 |
| **O** | slide 总览网格 | 主演示窗口 |
| **T** | 切换主题 | 主演示窗口 |
| **A** | 在当前 slide 循环演示一个动画 | 主演示窗口 |
| **F** | 全屏 | 主演示窗口 |

### URL 参数

| 参数 | 效果 | 示例 |
|------|------|------|
| `#3` 或 `#N` | 深链到第 N 页 | `deck.html#3` |
| `?preview=N` | 预览模式（只显示单页，隐藏 chrome） | `deck.html?preview=2` |

## 逐字稿编写指南

### 三条铁律

#### 铁律 1：提示信号，不是讲稿

逐字稿是你的**安全网**，不是要一字不差念的剧本。

- ❌ **差的写法** — 完整书面语，像论文摘要
- ✅ **好的写法** — 关键词加粗，过渡句独立成段

```
【差的逐字稿】
因此，我们引入了微服务架构，通过将单体应用拆分为多个
独立部署的服务单元，实现了系统的水平扩展能力，同时降低
了各模块之间的耦合度。

【好的逐字稿】
我们做了个决定 —— 拆分。

把**单体应用**拆成**多个独立服务**。

好处就两个词：**能扩容**、**低耦合**。

以前改一个地方，全站要重发。现在？**各管各的**。
```

#### 铁律 2：每页 150–300 字

约 **2–3 分钟/页** 的节奏。

- 少于 150 字 → 这页内容太薄，考虑合并
- 多于 300 字 → 信息过载，拆成两页

#### 铁律 3：用口语，不用书面语

| 书面语 ❌ | 口语 ✅ |
|-----------|---------|
| 因此 | 所以 |
| 该模块 | 这个模块 |
| 综上所述 | 总结一下 |
| 由此可见 | 你看 |
| 鉴于以上原因 | 所以为什么 |
| 不难发现 | 很明显 |

```
【书面腔 — 别这样写】
不难发现，该方案在性能方面取得了显著提升。

【口语化 — 这才是人话】
你看，性能直接上去了。
```

### 过渡句写法

过渡句是逐字稿的**骨架**，帮你从一页滑到下一页不掉链子。

**公式：承接上句 + 抛出下句**

```markdown
<!-- 从"问题"页过渡到"方案"页 -->
问题摆在这儿了。**那怎么破？** 来看看我们的方案。

<!-- 从"数据"页过渡到"结论"页 -->
数字不会骗人对吧？所以**结论很清晰** ——

<!-- 从"技术"页过渡到"业务"页 -->
技术层面的东西先聊到这。**回到业务上**，这带来了什么价值？
```

### 关键词加粗示例

```markdown
# 性能优化方案

**三个动作**，带来**十倍提升**。

第一，**加缓存** —— 热点数据 Redis 扛住。
第二，**异步化** —— 写操作丢到消息队列。
第三，**懒加载** —— 不需要的数据，就不查。

> 记住这个顺序：缓存 → 异步 → 懒加载。
> 按这个来，性能不会差。

```

> 💡 演讲时你的眼睛会自然扫到加粗部分 — 这就是提示信号的作用。
