# Frontend Design Orchestrator Core Contracts

本文档定义 `frontend-design-orchestrator` 技能的核心契约。该文档只定义安装路径、调用契约、输出产物契约和评估契约，不创建任何实际技能文件。

## Contract

### installation_path:
`~/.config/opencode/skill/frontend-design-orchestrator/`

- 该路径是唯一允许的安装目标路径。
- 技能根目录名称必须为 `frontend-design-orchestrator`，不得使用别名、缩写目录名或项目内相对路径替代。
- 所有后续实现、评估和打包动作都以该目录作为 `--skill-path` 指向目标。

### invocation:
当用户要求 AI 生成前端设计方案、输出高质量 `DESIGN.md`、整理页面/模块的视觉设计规范、补充图标系统说明、或基于 Vercel 设计指南执行设计审查时，必须触发该技能。

**输入触发条件**
- 用户明确提到“前端设计”“设计文档”“DESIGN.md”“设计规范”“设计审查”“Vercel 设计指南审查”等语义。
- 用户请求的目标是产出设计方案或审查结果，而不是直接编写业务实现代码。
- 用户提供的输入可以是以下任一形式，技能必须兼容：
  - 自然语言需求描述
  - 页面、模块、组件或产品目标说明
  - 已有 PRD、线框、截图、草图、设计片段或上下文链接
  - 需要被审查的现有设计文档或前端界面说明

**输入格式契约**
- 技能输入必须至少包含“设计对象”与“设计目标”两类信息；若用户未显式给出，技能应先从上下文提取，再在输出中注明采用的假设。
- 输入允许附带约束条件，例如品牌风格、平台范围、响应式要求、可访问性要求、组件约束或设计参考来源。
- 输入中若同时存在“生成设计文档”和“审查设计方案”两类目标，技能必须同时产出生成物与审查物，不得二选一省略。

**输出行为契约**
- 若任务以生成设计方案为主，主输出必须是结构化的 `DESIGN.md`。
- 若任务包含审查目标，除 `DESIGN.md` 外还必须输出审查产物。
- 技能输出只面向设计编排与审查，不负责生成业务代码、token 映射文件或图标清单文件。

### artifacts:
技能输出产物仅允许包含以下两类：

1. `DESIGN.md`
   - 这是主生成产物，必须生成。
   - 内容必须服务于前端设计落地，至少覆盖目标、信息架构或页面结构、视觉与交互原则、组件/模块设计要求、图标系统说明、响应式/可访问性要求，以及需要实现方遵循的约束。
   - `DESIGN.md` 是面向实现和评审的最终设计文档，不是头脑风暴草稿。

2. review artifacts
   - 这是审查产物集合，在任务包含“审查、review、audit、对照 Vercel 指南检查”等目标时必须生成。
   - 审查产物可以是 Markdown 文档、结构化审查条目或审查结论清单，但必须明确列出：检查维度、发现的问题、风险级别或影响、以及建议修改方向。
   - 审查产物必须显式说明其依据包含 Vercel 设计指南。

**明确排除项**
- 不输出 token mapping、design token 映射表、token 对照文件或任何 token 映射产物。
- 不输出 icon inventory、图标清单、图标资产目录或逐个图标枚举列表。
- 不输出业务实现代码、组件源码或样式源码，除非未来另有独立契约补充。

### evaluation_command:
`python -m skill_creator.scripts.run_eval --eval-set <path> --skill-path <path>`

- 该命令是唯一指定的评估命令格式。
- `--eval-set <path>` 必须指向本技能的评测集路径。
- `--skill-path <path>` 必须指向 `~/.config/opencode/skill/frontend-design-orchestrator/`。
- 评估时必须先建立 baseline，再用候选版本与 baseline 对比；没有 baseline 的评估结果无效。
- 评估关注点必须覆盖契约符合性：是否正确触发、是否生成 `DESIGN.md`、是否生成所需审查产物、是否避免 token 映射与图标清单、以及是否满足 Vercel 指南审查要求。

### pass_threshold:
候选技能版本只有在同时满足以下两个条件时才算通过：

1. 场景通过率 `>= 80%`
   - 通过率以 `run_eval` 输出中的场景级或 expectation 聚合结果为准。
   - 若通过率低于 `80%`，即使部分关键案例表现良好，也判定为不通过。

2. 相比 baseline 的通过率增量 `>= 20%`
   - 增量定义为：候选版本通过率减去 baseline 通过率。
   - 若候选版本通过率达到 `80%`，但相对 baseline 提升不足 `20%`，仍判定为不通过。

**判定规则**
- 采用 AND 关系，不是 OR 关系。
- 必须同时满足“绝对通过率阈值”和“相对 baseline 增量阈值”。
- 任一条件缺失、未记录或无法复现时，评估结论视为无效。

## Non-negotiable Boundaries

- 本契约文档只定义核心契约，不定义具体 `SKILL.md` 内容。
- 本契约文档不授权生成任何额外安装路径、别名命令或模糊产物名称。
- 后续实现若与本契约冲突，以本契约为准，必须先更新契约再变更实现。
