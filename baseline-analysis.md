# Baseline Analysis

## 分析方法

- 事实来源：6 个 `baseline-runs/*/outputs/assistant-response.md` 输出、`C:\Users\HP\.config\opencode\skill\skill-creator\agents\analyzer.md`、`writing-skills` 中关于合理化借口表的章节。
- 判定准则：按场景目标检查输出是否命中预期交付物、是否保持在设计/评审边界内、是否覆盖关键章节、是否出现用“确认后再做”“直接给代码”等方式跳过任务边界的合理化。
- 失败模式标签：`missing-sections`、`wrong-focus`、`scope-drift`、`tool-misuse`。

## 1. new-project-minimal-design-doc

### 期望

为登录页生成可直接用于设计对齐的设计文档，至少应稳定覆盖目标、布局、状态、交互、视觉，并把关键未决项压缩到少量可决策问题，而不是把核心设计继续外包给“后续确认”。

### 实际失败

- 输出整体是设计文档，说明模型理解了“要写文档”。
- 但真正的失效点出现在 `assistant-response.md:180-199`：最后列出 10 个待确认问题，其中包含第三方登录、验证码策略、品牌标识、响应式、埋点等核心设计输入，说明文档没有把关键约束沉淀成可执行结论，只是把不确定性原样往后推。
- 文档也缺少设计决策依据、假设前提、内容策略、无障碍细则等标准章节，属于“看起来完整，实际上还不能稳交付开发”。

### 逐字理性化解释

- `assistant-response.md:141`：`或打开忘记密码弹窗（需确认）`
- `assistant-response.md:186`：`第三方登录：具体支持哪些平台？图标样式？`
- `assistant-response.md:198-199`：`密码加密：前端传输是否需要加密？`、`埋点需求：需要统计哪些行为数据？`

这些表述的共同问题不是“提了问题”，而是把本该进入设计文档的关键约束包装成“待确认”，相当于用未决清单替代了设计决策。

### 识别模式

- `missing-sections`：缺少设计依据、假设、内容策略、无障碍细则、非功能约束。
- `wrong-focus`：把“补齐设计输入”退化成“罗列待确认问题”。

### 对技能设计的启示

技能需要明确区分“允许保留的开放问题”和“必须在本轮文档内收敛的关键设计约束”，否则模型会用一长串“待确认”伪装完成度。

## 2. new-project-with-reference-sketch

### 期望

基于参考草图产出首页设计文档，重点应该是把草图中的结构、层级、交互假设和未标注处的推断写清楚，而不是滑向技术实现建议。

### 实际失败

- 输出抓住了首页骨架，但对草图的利用偏浅，更像通用 landing page 模板。
- 首个明显失焦点在 `assistant-response.md:184-194`：文档直接给出 `React / Vue / 纯 HTML`、`Tailwind CSS / CSS Modules / Styled Components`、`Lucide / Heroicons / 自定义 SVG` 这类技术选型建议，已经从“设计编排”滑到“实现栈建议”。
- 同时，大量关键内容都被写成“需确认”，例如导航项、主标题、按钮文案、视觉区内容、卡片内容，说明输出没有把参考草图转译成足够具体的设计产物。

### 逐字理性化解释

- `assistant-response.md:52`：`主视觉插图/产品展示图（需确认内容形式）`
- `assistant-response.md:186-189`：`框架：React / Vue / 纯 HTML`、`样式方案：Tailwind CSS / CSS Modules / Styled Components`、`图标库：Lucide / Heroicons / 自定义 SVG`
- `assistant-response.md:198`：`等待确认后，可进入前端实现阶段。`

这里的理性化路径很典型：先用“需确认”保留设计空洞，再用“技术实现建议”制造一种已经推进到下一阶段的错觉。

### 识别模式

- `wrong-focus`：从草图解读转向技术栈建议。
- `scope-drift`：开始为前端实现阶段做铺垫，而不是把设计文档写扎实。
- `missing-sections`：缺少基于草图的内容假设、设计依据、关键决策说明。

### 对技能设计的启示

技能要明确要求“每个从草图推断出的内容都要标注依据或假设”，并禁止在设计阶段输出框架/样式方案等实现细节。

## 3. review-existing-design-md

### 期望

对现有 `DESIGN.md` 做结构化审查，既要覆盖章节完整性，也要覆盖可执行性、模糊点、风险优先级和阻塞项，输出应服务于“怎么补文档”，而不只是给分。

### 实际失败

- 这份 baseline 比其他场景更接近正确方向，已经有结构、交互、状态、视觉、可实现性和风险几个维度。
- 但失效点在于它把审查压成了“评分型报告”，而不是“补齐文档的精确清单”。`assistant-response.md:5`、`24`、`50`、`80`、`115`、`178` 多次出现打分，说明输出把重心放在评级展示，而不是形成可直接执行的修订框架。
- 同时缺少对原文引用、设计依据溯源、章节级修订建议、验收标准映射等审查关键件，导致“知道哪里差”，但“不足以直接返修”。

### 逐字理性化解释

- `assistant-response.md:5`：`评分：⭐⭐⭐ (3/5)`
- `assistant-response.md:178`：`综合评分：⭐⭐ (2.2/5)`
- `assistant-response.md:203`：`文档作为设计意图记录具有价值，但作为开发输入仍需大量补充。`

这里的理性化是：用评分和结论性话术替代逐章节、逐问题的修订指导，显得像“完成了审查”，实际却没有把返工路径写透。

### 识别模式

- `wrong-focus`：偏向打分展示，而不是返修导向的设计审查。
- `missing-sections`：缺少逐章节修订建议、引用依据、验收标准映射。

### 对技能设计的启示

技能应限制“评分型空转”，强制审查输出包含：发现位置、影响、建议补写内容、优先级和阻塞关系，避免模型用漂亮评分代替可执行反馈。

## 4. resolve-icon-system-conflict

### 期望

在 Lucide、Heroicons、自定义 SVG 的冲突中做设计系统层面的主方案决策，并说明决策依据、适用边界、迁移规则和例外条件；重点是“为什么这个体系适合当前设计系统”。

### 实际失败

- 输出在 `assistant-response.md:3` 一开头就给出 `推荐方案：Lucide Icons 作为主图标库`，先下结论、后补论据。
- 后面的比较表虽然列了数量、维护成本、生态支持等，但这些更偏工程便利性，而不是当前设计系统的视觉目标、现有资产兼容度、品牌表达需求。
- `assistant-response.md:67-71` 甚至直接落到了路径命名和元数据管理规范，说明输出部分精力花在资产管理细则上，而没有把“为什么是这套图标系统”讲得更扎实。

### 逐字理性化解释

- `assistant-response.md:3`：`推荐方案：Lucide Icons 作为主图标库`
- `assistant-response.md:17`：`覆盖率最高，减少 80% 的自定义需求`
- `assistant-response.md:67-71`：`存放路径：/assets/icons/custom/`、`命名：icon-{name}-{size}.svg`、`每个图标需标注：用途、来源、创建日期`

理性化路径是：用“覆盖率高、维护成本低”这样的工程效率理由，快速跳过设计系统冲突里最难的部分——视觉语言与品牌一致性的论证。

### 识别模式

- `wrong-focus`：偏向工程效率和资产管理，而不是设计系统决策依据。
- `scope-drift`：从选型论证滑到文件命名和目录规范。
- `missing-sections`：缺少与现有界面风格、品牌表达、混用成本相关的决策章节。

### 对技能设计的启示

技能需要明确要求“先比较设计语言，再比较工程成本”，并要求输出包含拒绝其他方案的理由，否则模型会默认选生态更熟的库。

## 5. negative-implementation-request

### 期望

这是负面场景：面对“直接实现页面”的请求，应拒绝写实现代码，并把用户拉回设计文档或设计评审边界。

### 实际失败

- 失败从第一行就开始：`assistant-response.md:1` 直接进入实现模式。
- 后续 300 行全部是 React + Tailwind 页面代码与交互实现，没有任何边界提醒、拒绝或重定向。
- 这不是轻微偏差，而是完全反着做：场景要求测试“是否能拒绝实现请求”，baseline 则把实现铺满了整份输出。

### 逐字理性化解释

- `assistant-response.md:1`：`我直接给你完整的 React + Tailwind 实现代码：`
- `assistant-response.md:291-299`：`实现要点：`、`交互实现：`、`移动端导航折叠（响应式菜单）`

这类话术把“用户想看结果”偷换成“可以直接越过设计阶段写代码”，属于最典型的越界合理化。

### 识别模式

- `wrong-focus`：把设计任务做成实现任务。
- `scope-drift`：完全漂移到 React + Tailwind 交付。

### 对技能设计的启示

技能必须把“禁止直接实现”写成显式红线，并提供替代动作模板，否则模型会用“直接给你完整代码”来表现积极配合。

## 6. negative-unrelated-backend-api-fix

### 期望

这是另一个负面场景：模型应该识别“后端 API 修复”与前端设计文档工作流无关，因此要拒绝处理或明确说明不属于当前范围。

### 实际失败

- 失败同样从第一段开始：`assistant-response.md:1-20` 直接定位后端 Bug 并给修复方案。
- 输出没有任何“这不是设计任务”的边界识别，而是完整进入 API 调试语境。
- 因为场景要求测试“是否会拒绝无关请求”，所以这个 baseline 属于整题失守。

### 逐字理性化解释

- `assistant-response.md:1`：`## 问题定位`
- `assistant-response.md:9-13`：`API 应该返回订单详情，而非单独的金额字符串`
- `assistant-response.md:15`：`## 修复方案`

这里的理性化非常直接：看到代码问题就进入修 bug 模式，完全忽略了当前工作流的主题和边界。

### 识别模式

- `wrong-focus`：把无关的后端调试当成当前任务。
- `scope-drift`：从前端设计工作流漂移到 API 修复。

### 对技能设计的启示

技能需要加入“无关请求识别”规则：只要问题核心不属于设计文档/设计评审/设计系统决策，就要明确拒绝并说明原因。

## 跨场景失败模式汇总

### 1. missing-sections

- 主要出现在 `new-project-minimal-design-doc`、`new-project-with-reference-sketch`、`review-existing-design-md`、`resolve-icon-system-conflict`。
- 常见症状：有结构，但缺少设计依据、假设前提、返修指令、验收标准、品牌/现状约束。

### 2. wrong-focus

- 6 个场景全部出现。
- 常见症状：把设计工作做成技术选型、评分秀、实现代码、后端修复。

### 3. scope-drift

- 在 `new-project-with-reference-sketch`、`resolve-icon-system-conflict`、`negative-implementation-request`、`negative-unrelated-backend-api-fix` 最明显。
- 常见症状：从设计编排漂移到技术栈建议、资产目录规范、完整实现、API 调试。

### 4. tool-misuse

- 从 `assistant-response.md` 无法直接证明具体工具调用错误，因此本轮只记录“未见可归责证据”。
- 这本身也是一个信号：如果技能希望模型必须读取输入文件、引用原文或显式说明边界，就应把这些动作写成可观测输出约束，否则事后难以判断工具纪律是否被遵守。

## 对技能设计的总体启示

1. **把边界写成红线，而不是建议。** 两个负面场景都说明：如果不显式禁止实现代码和无关请求，模型会默认“先解决问题再说”。
2. **把允许留白的范围写死。** 正向场景里，模型很喜欢用“需确认”“待确认”把关键设计约束推出去；技能需要规定最多保留多少个开放问题，以及哪些类型绝不能留白。
3. **强制“发现位置 + 影响 + 建议动作”三联输出。** 这能抑制评分型空转，也能让审查真正可执行。
4. **要求先论证设计依据，再给方案。** 图标冲突场景说明，如果不要求先比较视觉语言、品牌一致性和现状兼容性，模型会本能选生态最熟的库。
5. **把常见合理化原话直接写进技能。** `writing-skills` 的方法是对的：把 baseline 中出现过的借口逐字收编，再逐条反制，技能才会越测越稳。
