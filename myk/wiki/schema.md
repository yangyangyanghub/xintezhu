# Wiki Schema

> 本文件是 LLM Wiki 的核心配置。它定义 wiki 的结构、约定和工作流程。
> LLM 读取此文件后，应成为纪律严明的 wiki 维护者，而非通用聊天机器人。
> 你和 LLM 随着使用共同演化此文件。

---

## 1. 目录结构

```
myk/wiki/
├── index.md           # 全局内容索引（按分类列页+摘要+元数据）
├── log.md             # 操作日志（append-only，记录所有操作）
├── schema.md          # 本文件
├── topics/            # 主题页（按领域组织）
├── concepts/          # 概念页（跨领域术语/技术概念）
└── synthesis/         # 综合页（多个源之间的交叉分析/对比/决策）
```

### 各层含义

| 目录           | 内容                        | 谁写                    | 谁读      | 举例                        |
| ------------ | ------------------------- | --------------------- | ------- | ------------------------- |
| `topics/`    | 某个领域/主题的**全景概览**          | LLM（基于原始文档提取整合）       | LLM + 人 | `AI-Agent-生态.md`          |
| `concepts/`  | 独立的**概念/术语/协议**说明         | LLM（从文档中提取新概念时创建）     | LLM + 人 | `MCP-协议.md`               |
| `synthesis/` | 多个源之间的**对比/综合/决策**        | LLM（回答问题或 ingest 时生成） | LLM + 人 | `Multica-vs-Hermes-对比.md` |
| `myk/调研笔记/`  | **调研原始文档**（只读，LLM 绝不修改）   | 人                     | LLM + 人 | 调研报告、产品调研                 |

## 2. 页面格式约定

### 2.1 Frontmatter（每个页面必须有）

```yaml
---
title: "页面标题"
created: 2026-04-13
updated: 2026-04-13
sources: ["../调研笔记/xxx.md", "../技术沉淀/yyy.md"]  # 引用的原始文档
tags: [AI, Agent, 调研]  # 自由标签
status: active  # active | archived | outdated
---
```

### 2.2 内容结构（推荐但不强制）

```markdown
# 页面标题

> 一句话概述（用于索引和搜索）

## 核心要点
- 要点 1
- 要点 2
- 要点 3

## 详细内容

### 子标题
内容...

## 相关页面
- [[AI-Agent-生态]]
- [[MCP-协议]]

## 来源
- [[../调研笔记/xxx.md|调研报告标题]]
```

### 2.3 双链规范

- wiki 页面之间使用 Obsidian 标准双链 `[[页面名]]`（仅文件名即可，无需路径）
- 引用原始文档使用 `[[路径/文件名|显示名]]` 语法（必须带路径和别名）
- 概念首次在页面中出现时，创建双链指向 `concepts/` 中的概念页
- **没有对应概念页时，新建一个概念页而不是只写双链**
- 🔴 **禁止悬挂引用**：页面底部的"相关页面"中，引用的每一页必须是实际存在的 wiki 页面；若页面被删除，必须同步清理所有引用处

### 2.4 标签体系（Tags）

页面 frontmatter 中的 `tags` 字段应使用以下分类层级（可自由组合）：

```
领域层：教育 | GIS | 遥感 | LLM | Agent | 开发工具 | 前端 | 数据库 | 政策
功能层：评估 | 调研 | 提示词 | 模板 | 选型决策 | 规范 | 对比
技术层：BM25 | QGIS | MCP | Gateway | SPI | Processing | Atlas
```

示例：`tags: [教育, 评估, 优质均衡, 指标体系]`

### 2.5 页面命名规范

- **优先中文**：页面名用中文，便于搜索和双链引用
- **跨领域术语可保留英文**：如 `LLM-Gateway-选型-New-API-vs-LiteLLM`（技术名词保留英文）
- **稳定优先**：已有页面文件名一旦落盘，优先保持稳定；需要更自然展示名时，使用 `[[真实文件名|展示名]]`
- **使用连字符分隔**：英文单词间用 `-` 连接（kebab-case），不使用下划线
- **避免过长**：页面名控制在 30 字以内，超出部分在 frontmatter 的 `title` 字段中补充

推荐示例：
- `教育资源评估体系` ✅
- `LLM-Gateway-选型-New-API-vs-LiteLLM` ✅（文件名）
- `[[LLM-Gateway-选型-New-API-vs-LiteLLM|LLM Gateway 选型：New-API vs LiteLLM]]` ✅（显示名）
- `GIS专业制图提示词` ✅

可接受示例：
- `Harness-Engineering-技能工程` ✅（已落盘文件名，允许保留）
- `BM25-领域检索引擎` ✅（文件名带连字符时，正文可用 alias 显示自然标题）

## 3. 工作流程

### 3.1 Ingest（摄入新源）

**触发条件**：新增调研报告 / 技术文档 / 文章 / PDF

**执行步骤**：

1. 读取新源，提取关键观点、数据、结论
2. 与洋哥讨论确认重点（如果需要）
3. 创建或更新 `topics/` 中对应主题页
4. 识别新概念时创建 `concepts/` 页面
5. 如果新源与已有源有**对比/矛盾/补充**关系，创建或更新 `synthesis/` 页面
6. 更新 `index.md`（添加条目+一行摘要）
7. 更新 `log.md`（追加 ingest 记录）

**单个源可能影响 10-15 个 wiki 页面，这是正常的。**

### 3.2 Query（查询）

**触发条件**：洋哥提问、需要综合多份文档的回答

**执行步骤**：

1. **优先查询 wiki**（不是直接读原始文档）
2. 读取相关页面，综合回答
3. 回答中产生的新见解（对比/总结/决策），**写回 wiki 作为新页面**
4. 在 `log.md` 追加 query 记录

### 3.3 Lint（健康检查）

**触发条件**：洋哥说"检查 wiki" / 每周自动执行

**执行步骤**：

1. 检查**矛盾**：不同来源对同一概念的描述是否冲突？标注冲突
2. 检查**过时**：新源是否让旧源的内容失效？标记 outdated
3. 检查**孤立页面**：无入链的页面（或入链 < 1），建议合并或补充
4. 检查**缺失概念**：内容中提到了重要概念但没有概念页，建议新建
5. 建议**新调研方向**：基于知识 gaps，建议洋哥补充哪些来源
6. 在 `log.md` 追加 lint 记录

## 4. Index 和 Log 规范

### 4.1 index.md

- **定位**：内容导向，全局目录
- **格式**：按分类组织（topics / concepts / synthesis），每行一个条目
- **每条内容**：`[[页面路径]] — 一句话摘要`
- **更新时机**：每次 ingest / 新建页面时
- **规模预期**：~100 个条目时仍可用，超过后考虑按子目录拆分

### 4.2 log.md

- **定位**：时间线，记录所有操作历史
- **格式**：每条以 `## [YYYY-MM-DD] action | 标题` 开头（可被 Unix 工具解析）
- **action 类型**：`ingest` / `query` / `lint` / `update` / `archive`
- **追加规则**：只追加，不修改已有条目
- **查询最近操作**：`grep "^\#\# \[" log.md | tail -5`

## 5. 质量准则

作为 wiki 维护者，你（LLM）应该：

- ✅ 每次 ingest 后更新所有受影响的页面，不要只改一页
- ✅ 发现矛盾时在页面中标注 "⚠️ 与 [[另一页面]] 矛盾"
- ✅ 保持页面简洁，详细内容放在子页面
- ✅ 每条更新都体现在 index.md 和 log.md 中
- ❌ 不要修改 `myk/调研笔记/` 和 `myk/技术沉淀/` 中的任何文件
- ❌ 不要在 wiki 中重复粘贴原始文档全文（应该提取、整合、链接）
- ❌ 不要创建没有入链的孤立页面

## 6. 与 .memory/ 的关系

| 系统 | 定位 | 存储 | 内容 |
|------|------|------|------|
| `.memory/` | LLM 的记忆（身份、习惯、对话摘要） | LLM 管理 | 工作习惯、偏好、流程 |
| `myk/wiki/` | 人+LLM 共建的知识库 | 文件系统 | 调研结论、技术知识、决策记录 |

两者互补但不重合：
- `.memory/` = **怎么做**（流程、习惯、偏好）
- `myk/wiki/` = **是什么**（知识、结论、对比）
