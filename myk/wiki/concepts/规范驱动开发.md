# 规范驱动开发（Spec-Driven Development）

> 一种软件开发方法论，要求在编写代码前先创建规范文档（spec），通过规范定义需求、约束和验收标准，确保开发过程有据可循。

## 核心要点
- **先规范后编码**：核心原则是"先写规范，再写代码"，避免无边界开发
- **工件驱动**：通过 proposal → spec → plan → tasks 的工件链确保可追溯性
- **AI 友好**：特别适合 AI Agent 编码场景，防止 AI 超出需求范围
- **框架实现**：代表工具有 GitHub Spec-Kit、Fission-AI OpenSpec、Superpowers

## 详细内容

### 核心理念
规范驱动开发源于一个核心问题：在 AI 编程时代，如何让 AI 真正理解意图并按预期工作？

SDD 将需求转化为"工件"（artifacts），代码变更只在工件定义的边界内进行。工件生命周期通常包含：
1. **宪法/约束**（constitution）：项目级全局原则
2. **功能规范**（spec）：描述 what 和 why，不涉及技术实现
3. **技术计划**（plan）：技术栈选择、架构设计、API 契约
4. **任务清单**（tasks）：可执行的具体步骤
5. **实现**（implement/apply）：按计划执行代码变更
6. **验证**（verify）：检查实现是否与规范对齐
7. **归档**（archive）：将完成的变更归入知识库

### 与 Harness 的关系
[[Harness-Engineering-技能工程]] 是规范驱动开发的具体应用模式。Harness 将 SDD 与 [[MCP-协议]]、[[Agent-as-Teammate]] 等概念结合，形成"结构化约束 + 评测闭环"的工程实践，让 AI 编码成功率从 80% 提升到 98.2%。

### 适用场景
- 有一定历史遗留的业务系统增量改造
- 团队希望把 AI 编码变成"流程能力"而非个人技巧
- 需要将"需求、实现、评审、校验"明确拆分的场景
