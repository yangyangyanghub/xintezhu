# 上下文压缩

> 针对 LLM 会话上下文膨胀问题的压缩技术与记忆系统结合方案

---

## 核心概念

**上下文压缩（Context Compression）**：在不丢失关键信息的前提下，将会话中的原始内容压缩为更紧凑的表示形式，从而控制 token 消耗、降低延迟、避免上下文窗口溢出。

---

## 为什么需要上下文压缩

### 问题根源

| 问题 | 表现 | 影响 |
|------|------|------|
| 上下文窗口有限 | 系统提示 + 对话历史 + Tool定义 + Skill 说明 + 记忆文件共同争抢 | 超出上限后模型"失忆"或报错 |
| Token 消耗快 | 长对话中 90% 以上是冗余上下文 | 成本飙升 |
| 跨会话记忆丢失 | 新会话是一张白纸，需重复说明背景 | 效率极低 |
| 压缩质量差 | 简单截断代替语义压缩 | 关键决策、踩坑记录丢失 |

---

## OpenClaw 当前的上下文压缩机制

### Compaction 机制

当对话接近上下文窗口上限时触发：

```
上下文使用率达阈值 → 触发 memoryFlush → Agent 保存重要信息到记忆文件
                    → 将早期消息压缩为摘要 → 释放上下文空间 → 继续对话
```

**问题**：
- compaction 本质是**截断**，不是真正的语义压缩
- memoryFlush 依赖 Agent 自觉性，可能遗漏关键信息
- MEMORY.md 全量注入，记忆多了撑爆上下文，少了丢失上下文
- 每日日志是原始流水账，没有压缩

### 配置参数

```json
{
  "compaction": {
    "reserveTokensFloor": 20000,
    "memoryFlush": {
      "enabled": true,
      "softThresholdTokens": 4000,
      "systemPrompt": "Session nearing compaction. Store durable memories now.",
      "prompt": "Write any lasting notes to memory/YYYY-MM-DD.md; reply with NO_REPLY if nothing to store."
    }
  }
}
```

## 主流上下文压缩技术路线

### 1. 会话内压缩（Session-Level）

| 技术 | 原理 | 优缺点 |
|------|------|--------|
| **Compaction 截断**（OpenClaw） | 旧消息替换为 summary message | 简单但丢失细节，仅会话内有效 |
| **分段摘要**（Claude Code） | summary.md 维护会话结构化摘要，compact 时 section 级裁剪 | 更结构化，但仍限会话内 |
| **Latent Context Compilation** | 用 disposable LoRA 将长上下文蒸馏为 compact buffer tokens | 16x 压缩率，保留细粒度细节，但需模型支持 |

### 2. 跨会话记忆压缩

| 技术 | 原理 | 优缺点 |
|------|------|--------|
| **claude-mem 三层检索** | search（索引）→ timeline（时间线）→ get_observations（详情） | 20000 → 3000 tokens，10倍节省 |
| **SimpleMem** | 语义无损压缩 + 在线语义合成 + 意图感知检索 | F1 提升 26.4%，token 消耗降 30 倍 |
| **ClawXMemory 多级记忆** | L0 原始会话 → L1 片段 → L2 项目记忆 → Profile | 模型驱动选择，类人推理 |
| **MemArt** | 历史存为 KV cache 块，注意力分数检索 | 省去重新编码，Acc 提升 11%+ |

### 3. Agent 场景压缩

| 技术 | 原理 | 优缺点 |
|------|------|--------|
| **Focus（Active Context Compression）** | Agent 自主决定何时合并关键知识、何时裁剪原始历史 | 22.7% token 节省，准确率不变 |
| **ACON** | 优化压缩 guideline，自然语言空间优化 | 内存降低 26-54%，蒸馏后保留 95% 精度 |
| **ATACompressor** | 任务感知自适应压缩，只压缩任务相关部分 | Q&A 性能优于基线 |

## 与记忆系统的结合方案

### 分级记忆压缩架构

```
L0 原始会话（完整上下文）
    ↓ compaction 触发
L1 记忆片段（结构化摘要：决策、变更、踩坑）
    ↓ 定期整合（topic shift / idle 触发）
L2 项目记忆（高频决策、核心约定、架构决策）
    ↓ 精炼维护
Profile 核心 MEMORY.md（2000 tokens 以内）
```

### 三级渐进式检索

```
新会话启动：
  → search（索引检索，~50-100 tokens/条）
  → 初步筛选相关记忆
  → timeline（时间线定位，确认上下文关联）
  → get_observations（按需拉取详情，~500-1000 tokens/条）

总注入量控制在 3000 tokens 以内
```

### Compaction 前自动压缩

优化 memory flush prompt：

```
Session nearing compaction. Before old messages are summarized,
extract and save:
1. Architecture decisions made and their rationale
2. Coding standards and project conventions established
3. Critical bug fixes and root causes identified
4. Uncompleted tasks and their current status
5. Any pattern or insight worth remembering for future sessions

Write structured notes to memory/YYYY-MM-DD.md. Reply NO_REPLY if nothing.
```

## 推荐实施路线

### 短期（零成本）

- 优化 compaction 前的 memoryFlush prompt，明确列出需要保存的信息类型
- 定期整理 MEMORY.md，保持 2000 tokens 以内
- 利用 OpenClaw 自带的 memory_search + memory_get 工具按需检索

### 中期（claude-mem 集成）

- 部署 claude-mem 插件（`npx claude-mem install`）
- 获得跨会话持久记忆 + 三层渐进式检索
- 20000 → 3000 tokens 节省，约 90% token 降低

### 长期（深度定制）

- 参考 ClawXMemory 多级记忆架构
- 结合 ACON 框架优化压缩 guideline
- 考虑 MemArt 方向的 KV cache 级原生压缩（需模型支持）
