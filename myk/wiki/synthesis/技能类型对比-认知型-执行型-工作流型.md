# 技能类型对比：认知型 vs 执行型 vs 工作流型

> AI Agent 技能生态的三种核心形态，各自解决不同维度的问题。

---

## 核心对比

| 维度 | 认知型（Cognitive） | 执行型（Execution） | 工作流型（Workflow） |
|------|---------------------|---------------------|---------------------|
| **解决什么** | 哪些活值得干、按什么顺序干 | 怎么把活干出来 | 怎么把活干规范 |
| **典型代表** | slavingia/skills | OpenCode 工具 Skills | superpowers |
| **绑定程度** | 低（纯 Markdown） | 高（依赖工具/命令） | 中（依赖脚本） |
| **世界观** | 强（单一方法论统一） | 弱（多来源组合） | 中（统一方法论） |
| **输出形式** | 决策建议、SOP、判断 | 代码、文件、API 调用 | 流程执行结果 |
| **迁移成本** | 低 | 高 | 中 |
| **适合场景** | 创业决策、战略规划 | 开发、浏览器、MCP 操作 | TDD、Debug、Code Review |

---

## 生态分层关系

```
┌─────────────────────────────────┐
│   上层：认知型 Skills            │  决策方向
│   slavingia/skills              │  "先做什么"
├─────────────────────────────────┤
│   中层：工作流型 Skills          │  质量保障
│   superpowers, TDD, Debug       │  "怎么规范做"
├─────────────────────────────────┤
│   中层：执行型 Skills            │  能力支撑
│   OpenCode Skills, MCP tools    │  "怎么做到"
├─────────────────────────────────┤
│   底层：标准与运行时             │  基础设施
│   Claude Skills, Agent Spec     │  "怎么加载"
└─────────────────────────────────┘
```

---

## 互补关系而非替代关系

三者不是互斥的，最佳实践是组合使用：

```
认知型 → 决定做什么（方向）
    ↓
工作流型 → 决定怎么做规范（质量）
    ↓
执行型 → 决定怎么做到（能力）
```

**示例：一人公司创业流程**

1. 用 `slavingia/skills` 的 find-community 决定服务哪个社区
2. 用 `superpowers` 的 TDD 工作流规范产品开发
3. 用 OpenCode Skills 的代码生成能力实际编写代码
4. 用 `slavingia/skills` 的 pricing 决定定价策略
5. 用 `superpowers` 的 code-review 规范审查代码质量

---

## 趋势判断

1. **认知型将是差异化最大的层面**：执行型会趋同（工具能力大家都差不多），但认知型的世界观可以千差万别
2. **编写 skills 的人将比调用 skills 的人更有价值**：前者定义规则，后者执行规则
3. **产品化方法论将成为知识付费新形态**：书籍/课程 → skill 包，可被 Agent 直接消费
