# 院内 AI 模型成本治理

> 院内 AI 成本治理的核心不是“换更便宜模型”这么简单，而是先识别真实负载结构：HDPD 当前是典型长输入短输出场景，对输入 token 价格极度敏感，因此应优先做缓存命中率治理，再考虑模型切换。

---

## 核心判断

2026-06-02 的模型选型汇报形成了一个优先级清晰的治理路径：

1. **先做缓存治理**：给请求增加 `cache_control` 标记，利用百炼显式缓存机制，优先降低输入 token 成本。
2. **再做灰度验证**：用 1 周 POC 和 2 个调用方灰度，获取真实命中率数据。
3. **最后再决定是否换模型**：DeepSeek V4-Flash 是长期降本备选，但不应跳过缓存方案直接迁移。

一句话结论：

> 加一行 `cache_control` 标记 → 月省 ¥1,800-5,200 → 零迁移、低风险、可回滚。

## 现状数据

| 指标 | 数值 | 治理含义 |
|---|---:|---|
| 日均输入 | 73M tokens | 输入成本是主要矛盾 |
| 日均输出 | 0.37M tokens | 输出成本影响较小 |
| 输入输出比 | 198:1 | 典型长输入短输出场景 |
| 平均 prompt | 34K tokens | 重复上下文具备缓存价值 |
| 最大 prompt | 489K tokens | 模型必须支持长上下文 |
| 当前缓存命中率 | 0% | 存在直接优化空间 |
| 理论月成本 | ¥8,880 | 用于测算按量价上限 |

## 方案优先级

| 方案 | 月成本/节省 | 改造量 | 定位 |
|---|---:|---|---|
| 方案 0：加缓存标记 | 节省 ¥1,800-5,200 | 约 1 行代码/中间件注入 | 立即做，低风险 |
| 方案 B：DeepSeek V4-Flash | 理论约 ¥1,840/月 | 需要切换厂商与业务灰度 | 长期降本备选 |
| 方案 A：阿里 Token Plan 直达 | 约 ¥1,898/月 | 零业务改造 | 不想动架构时兜底 |
| 方案 C：腾讯混元 + DeepSeek 混搭 | 约 ¥1,692/月 | 改造复杂 | 极致省钱备选 |

**决策原则**：方案 0 与后续任何方案不冲突，所以应先做。即使未来切 DeepSeek，缓存治理经验和请求结构治理仍然可复用。

## 缓存治理机制

百炼支持显式缓存，new-api 能透传相关字段，缺口主要在请求构造或中间件注入。

```text
长 system prompt / RAG 检索上下文
        ↓
增加 cache_control 标记
        ↓
相同前缀在 TTL 内命中缓存
        ↓
命中部分按 10% 计费
        ↓
输入 token 成本下降
```

实测验证：

- 第 1 次请求：冷启动，命中率 0%。
- 第 2 次请求：重复 prompt，命中率 99%（1889/1907 tokens）。
- 真实业务保守估计：命中率可能在 60-90%，需要灰度验证。

## 落地节奏

| 时间 | 动作 | 输出 |
|---|---|---|
| 2026-06-02 至 2026-06-06 | 改 2 个调用方并跑 48 小时 | 真实命中率数据 |
| 2026-06-09 | 汇报 POC 结果 | 决策：推广缓存 or 切方案 B |
| 2026-06-16 | 全院推广或启动 DeepSeek 灰度 | 月成本曲线与治理策略 |

建议灰度场景：

- RAG 文档问答。
- 合同审查。

这两类场景更容易出现固定 system prompt、固定文档背景或重复检索上下文，适合验证缓存收益。

## 风险边界

| 风险 | 边界控制 |
|---|---|
| 命中率不及预期 | 先 1 周灰度，不直接全院推广；低于阈值则启动模型切换评估 |
| 账单与理论测算不一致 | 理论值基于 token 用量和按量价，真实账单需结合 Token Plan 套餐核算 |
| 业务方效果不接受模型切换 | DeepSeek 仅作为 Plan B，必须通过典型场景盲测和 new-api 灰度 |
| 合规风险 | 推荐方案限定在国内厂商，生产数据不走第三方中转 |
| 回滚风险 | 优先在 new-api 中间件灰度注入，异常时 30 秒回滚 |

## 沟通策略

该主题对领导汇报时要避免把异常峰值归咎到具体个人或 token，应统一表述为“批量文档处理任务”。核心叙事应是：

- 这说明院内 AI 已经进入真实批量业务场景。
- 暴露的问题是架构对突发并发和成本治理支撑不足。
- 优化方向是提升系统承载力和成本可控性，而不是限制业务使用。
