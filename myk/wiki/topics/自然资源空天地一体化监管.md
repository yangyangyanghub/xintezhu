# 自然资源空天地一体化监管

> 自然资源低空监管不应被讲成单一无人机巡查应用，而应放进“时空数据底座 + 套合监管系统 + 卫星遥感 + 低空巡查 + 业务闭环”的县域自然资源监管体系中。

---

## 核心定位

推荐口径：

> 以时空数据智能开放平台为底座，以套合监管系统为核心能力，以卫星遥感和低空巡查为感知手段，构建县域自然资源空天地一体化智能监管体系。

低空影像解决“现场怎么看清”，但自然资源监管真正需要回答的是：

- 变化发生在哪里？
- 占了什么地类？
- 是否压占永久基本农田、生态保护红线、城镇开发边界？
- 是否有审批、是否超审批、是否少批多占？
- 责任主体是谁，归哪个乡镇、村、宗地或权利人？
- 后续如何核查、整改、复核、归档？

## 四梁架构

```text
一底座：时空数据智能开放平台
一核心：套合监管系统
一抓手：卫星遥感 + 低空智能巡查
一闭环：自然资源业务监管流程
```

### 一底座：时空数据智能开放平台

统一汇聚自然资源监管所需的底账、底图、底线、底数，包括：

- 国土变更调查、三调和年度变更。
- 永久基本农田、耕地保护目标、生态保护红线、城镇开发边界。
- 国土空间规划、建设用地审批、土地供应、规划许可。
- 执法监察图斑、设施农用地备案、临时用地审批。
- 卫星遥感影像、无人机低空影像、倾斜摄影。
- 行政区、村界、宗地、权属等基础数据。

### 一核心：套合监管系统

套合监管系统把影像发现的“现象”转成自然资源管理需要的“问题属性”。

| 套合对象 | 套合目的 | 输出判断 |
|---|---|---|
| 地类图斑 | 判断占用地类 | 耕地、建设用地、林地、园地、未利用地等 |
| 永久基本农田 | 判断是否触碰底线 | 是否占用永久基本农田 |
| 国土空间规划 | 判断是否符合规划用途 | 是否符合规划管控要求 |
| 城镇开发边界 | 判断建设管控范围 | 城镇集中建设区 / 边界外 |
| 生态保护红线 | 判断生态约束 | 是否压占生态保护红线 |
| 审批红线 | 判断合法审批依据 | 未批先建、超范围建设、少批多占 |
| 设施农用地备案范围 | 判断是否超备案或变用途 | 超范围建设、改变用途 |
| 临时用地范围 | 判断是否超期、超范围 | 临时用地违规线索 |
| 权属 / 宗地 | 判断责任主体 | 关联乡镇、村、单位、权利人 |

### 一抓手：卫星遥感 + 低空智能巡查

```text
卫星看全域
低空看细节
套合判属性
平台管闭环
```

- 卫星遥感：负责全域周期性变化发现。
- 低空巡查：负责重点区域精细核查、取证和复核。
- 套合监管：负责把变化图斑与管理数据叠加研判。
- 平台闭环：负责问题流转、整改处置、复核销号和台账归档。

### 一闭环：自然资源业务监管流程

```text
影像采集 → 智能识别 → 疑似图斑 → 自动套合 → 属性研判 → 人工核查 → 整改处置 → 低空复核 → 台账归档
```

## 核心应用场景

以临漳县为例，不设置矿山监管场景，重点保留 5 类自然资源主责主业场景：

| 场景 | 发现方式 | 套合判断 | 输出成果 |
|---|---|---|---|
| 耕地保护监管 | 卫星变化、低空巡查、田长制线索 | 是否占耕、是否占永久基本农田、是否存在非农化/非粮化风险 | 疑似占耕图斑、面积统计、位置清单、影像证据 |
| 违法用地监管 | 新增建设变化图斑、低空巡查、群众举报线索 | 是否有审批、是否超审批范围、是否符合规划、是否少批多占 | 未批先建、少批多占、违法占地线索清单 |
| 设施农用地监管 | 低空影像、备案数据、日常巡查 | 是否超备案范围、是否改变用途、是否硬化、是否存在“大棚房”风险 | 设施农用地问题台账、前后对比图、核查建议 |
| 建设项目批后监管 | 项目红线叠加低空影像、施工进度巡查 | 是否按审批范围建设、是否超红线、是否长期闲置、是否擅自改变用途 | 批后监管清单、项目进度影像、异常问题图斑 |
| 县城及村庄违建房监管 | 低空巡查、变化检测、重点区域航拍 | 套合地类、规划、审批、权属、历史台账 | 疑似违建点位、楼顶/院落扩建清单、拆后复核成果 |

## 低空智能监管平台建设要点

2026-06-10 对邯郸市低空智能监管平台现网评估后，进一步明确：低空平台不应只做“无人机指挥调度 + 数据展示”，而应补齐设备、任务、成果、告警、AI 识别和外部监管协同的闭环能力。

### 六条主线

| 主线 | 重点 | 易踩坑 |
|---|---|---|
| 设备底账 | 一机一档、一机一码、主编号和显示名称分离 | 把单位、点位、型号、用途混进设备编号 |
| 实时态势 | 在线状态、位置、高度、速度、航向、轨迹 | 在线、离线、任务中、位置有效口径不一致 |
| 任务管理 | 任务编号、任务类型、执行设备、航线、状态 | 任务名称临时化、测试化，无法复盘 |
| 成果资产 | 照片、视频、航迹、识别结果关联任务与设备 | 数据中心统计和列表口径不一致 |
| 空间套合 | 行政区、地类、规划、红线、审批、权属叠加 | 只看影像，不回到自然资源管理规则 |
| 问题闭环 | 派发、核查、整改、复核、销号、归档 | 只做 AI 识别，不做处置闭环 |

### 编码原则

邯郸市低空监管平台建议采用“一机一档、一机一码、多码映射、全程可追溯”的设备编码规则。平台主编号只标识设备身份，不承载单位、任务、乡镇或临时载荷；民航实名登记标志、唯一产品识别码、远程识别码、飞服平台设备 ID、公安备案号和厂商序列号均作为外部 ID 映射字段维护。

推荐主编号格式：

```text
HD-UAV-{区县行政区划代码}-{飞行器类别码}-{四位流水号}
```

## 表达边界

- 不把低空巡查讲成主角，主角是自然资源监管体系。
- 不承诺 AI 自动判定违法，只表达为辅助发现线索和辅助研判属性。
- 不把县城违建房监管讲成全部建设内容，它只是首个示范应用。
- 场景必须贴合地方实际；例如临漳县没有矿山，不设置矿山监管。
- 不把展示型大屏作为优先建设重点；底层设备底账、任务成果关系和问题闭环更关键。
