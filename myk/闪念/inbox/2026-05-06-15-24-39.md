## 🧠 核心发现

### 当前上下文压缩不完善的原因

1. **compaction 本质是截断，不是压缩**
   OpenClaw 的 compaction 机制只是把旧消息替换为一条 summary message，属于**会话内截断**。旧对话内容被丢弃而非结构化压缩，导致跨会话时信息丢失。

2. **memory flush 依赖 Agent 自觉性**
   当上下文快满时，系统触发 memoryFlush，让 Agent 自己判断什么值得持久化。但 Agent 可能遗漏关键信息，或写入冗余内容，缺乏质量把控。

3. **MEMORY.md 容量限制与注入方式粗暴**
   OpenClaw 建议 MEMORY.md 保持 2000 tokens 以内，每次会话启动时**全量注入**。记忆多了就撑爆上下文，少了又丢失上下文，缺乏按需注入能力。

4. **缺乏跨会话的语义压缩**
   每日日志（YYYY-MM-DD.md）是原始流水账，没有压缩。claude-mem 解决了这个问题，但 OpenClaw 原生没有跨会话的语义摘要机制。

### 与记忆系统的结合点

1. **compaction 前自动触发记忆压缩**
   在 compaction 前不只是提醒 Agent「存重要信息」，而是自动将即将被丢弃的上下文通过语义压缩提取关键决策、代码变更、踩坑记录，结构化写入记忆文件。

2. **三层渐进式检索按需注入**
   参考 claude-mem 的方案：search（索引检索，~50 tokens/条）→ timeline（时间线定位）→ get_observations（按需拉取详情，~500 tokens/条）。实现约 10 倍 token 节省。

3. **分级记忆压缩策略**
   - L0 原始会话 → 自动压缩为 L1 记忆片段（结构化摘要）
   - L1 记忆片段 → 定期整合为 L2 项目记忆（高频决策、核心约定）
   - L2 项目记忆 → 维护核心 MEMORY.md（控制在 2000 tokens 内）

4. **KV Cache 级别的原生压缩**
   前沿研究（MemArt）直接将历史对话存为 KV cache 块，通过注意力分数检索，省去重新编码的开销。这是更底层的压缩方向，但需要模型侧支持。

### 推荐方案

**短期（零成本）：**
- 优化 compaction 前的 memoryFlush prompt，明确列出需要保存的信息类型（架构决策、编码规范、未完成任务、关键 bug 修复）
- 定期整理 MEMORY.md，保持精简

**中期（claude-mem 集成）：**
- 部署 claude-mem 插件，获得跨会话持久记忆
- 利用三层渐进式检索实现按需注入，20000 → 3000 tokens

**长期（深度定制）：**
- 参考 ClawXMemory 的多级记忆架构（L0 → L1 → L2 → Profile）
- 结合 ACON 框架优化压缩 guideline，让压缩质量更高
