# 邯郸市低空监管平台无人机编码规范

## 1. 背景

随着邯郸市各县区逐步部署无人机，平台需要统一设备编号、显示名称、载荷管理、乡镇管理和外部监管 ID 映射规则。该规范用于支撑市县两级监管，并预留与公安、民航实名登记、远程识别、低空飞行服务平台的数据打通能力。

## 2. 总体原则

1. **一机一档，一机一码**：每架无人机建立独立设备档案，并生成一个平台内部主编号。
2. **主编号稳定**：编号只标识设备身份，不承载单位、责任人、任务用途、当前乡镇或临时载荷。
3. **多码映射**：平台编号、民航实名登记标志、唯一产品识别码、远程识别码、飞服平台设备 ID、公安备案号、厂商序列号分别维护，不相互替代。
4. **编号到县区，管理到乡镇**：主编号标识到区县；乡镇信息通过字段、任务记录和 GIS 空间匹配维护。
5. **编号管身份，字段管能力，任务管实际使用**：平台编号回答“这是谁”，载荷字段回答“它能干什么”，任务载荷回答“这次带了什么”。

## 3. 平台主编号规则

推荐格式：

```text
HD-UAV-{区县行政区划代码}-{飞行器类别码}-{四位流水号}
```

示例：

```text
HD-UAV-130402-MR-0001
```

字段说明：

| 字段           | 含义          | 示例       |
| ------------ | ----------- | -------- |
| `HD`         | 邯郸市标识       | `HD`     |
| `UAV`        | 无人机设备类型     | `UAV`    |
| `{区县行政区划代码}` | 设备登记归属区县    | `130402` |
| `{飞行器类别码}`   | 飞行器类别       | `MR`     |
| `{四位流水号}`    | 区县 + 类别下流水号 | `0001`   |

编号示例：

| 编号 | 含义 |
|---|---|
| `HD-UAV-130400-MR-0001` | 市级登记多旋翼无人机 1 号 |
| `HD-UAV-130402-MR-0001` | 邯山区登记多旋翼无人机 1 号 |
| `HD-UAV-130403-MR-0001` | 丛台区登记多旋翼无人机 1 号 |
| `HD-UAV-130481-VT-0001` | 武安市登记垂直起降固定翼无人机 1 号 |

## 4. 飞行器类别码

| 飞行器类别     | 代码   | 说明                  |
| --------- | ---- | ------------------- |
| 多旋翼       | `MR` | Multi-Rotor         |
| 固定翼       | `FW` | Fixed-Wing          |
| 垂直起降固定翼   | `VT` | VTOL                |
| 无人直升机     | `UH` | Unmanned Helicopter |
| 穿越机 / FPV | `FP` | FPV                 |
| 其他        | `OT` | Other               |

## 5. 显示名称规则

平台主编号用于系统唯一标识，显示名称用于业务人员识别。

推荐格式：

```text
乡镇简称-型号-主载荷/综合载荷-序号
```

示例：

```text
河沙镇-M350-热成像-01
代召乡-Mavic3E-可见光-01
南堡乡-M30T-综合载荷-01
县级中心-VTOL-倾斜摄影-01
```

显示名称可以随常驻地、主载荷或业务表达调整；平台主编号不变。

## 6. 载荷管理规则

### 6.1 载荷不进入主编号

载荷能力不建议写入平台主编号。原因是载荷可能更换，一架无人机可能具备多个载荷，任务实际挂载也可能与设备默认载荷不同。跨系统对接依赖法定身份字段，不依赖地方编号中的载荷语义。

### 6.2 载荷字段

| 字段                   | 含义         | 示例            |
| -------------------- | ---------- | ------------- |
| `primaryPayloadCode` | 主载荷代码      | `IR`          |
| `payloadCodes`       | 全部载荷能力     | `EO,IR,SP,SL` |
| `payloadConfigType`  | 载荷配置类型     | `MX`          |
| `payloadMountType`   | 挂载方式       | 固定挂载 / 可更换挂载  |
| `payloadChangeable`  | 是否可更换      | `true`        |
| `taskPayloadCodes`   | 本次任务实际挂载载荷 | `IR,SP,SL`    |

### 6.3 载荷代码表

| 载荷类型     | 代码   | 说明                  |
| -------- | ---- | ------------------- |
| 可见光相机    | `EO` | Electro-Optical     |
| 红外 / 热成像 | `IR` | Infrared            |
| 激光雷达     | `LR` | LiDAR               |
| 倾斜摄影     | `OB` | Oblique Camera      |
| 多光谱      | `MS` | Multispectral       |
| 高光谱      | `HS` | Hyperspectral       |
| 喊话器      | `SP` | Speaker             |
| 探照灯      | `SL` | Search Light        |
| 抛投器      | `DP` | Drop Payload        |
| 气体检测     | `GD` | Gas Detector        |
| 通信中继     | `CR` | Communication Relay |
| 喷洒 / 植保  | `PS` | Spraying            |
| 综合载荷     | `MX` | Mixed Payload       |
| 无载荷 / 训练 | `NO` | None                |
| 其他       | `OT` | Other               |

### 6.4 多载荷表达

不建议把多个载荷全部写入编号，例如：

```text
HD-UAV-130402-MR-EO-IR-SP-SL-0001
```

推荐表达：

```text
platformUavCode：HD-UAV-130402-MR-0001
displayName：河沙镇-M350-综合载荷-01
primaryPayloadCode：IR
payloadConfigType：MX
payloadCodes：EO,IR,SP,SL
```

设备档案中的载荷表示设备具备或登记的能力；任务载荷表示某次任务实际挂载的载荷。

## 7. 乡镇管理规则

县区部署时，县里需要看到无人机所在乡镇，但乡镇不应写入平台主编号。原因是无人机会跨乡镇飞行，常驻乡镇可能变化，乡镇行政区划也可能调整。

建议维护字段：

| 字段                   | 含义             |
| -------------------- | -------------- |
| `registeredTownCode` | 登记归属乡镇代码       |
| `registeredTownName` | 登记归属乡镇名称       |
| `homeTownCode`       | 常驻机库或起降点所在乡镇代码 |
| `homeTownName`       | 常驻机库或起降点所在乡镇名称 |
| `currentTownCode`    | 当前实时位置所在乡镇代码   |
| `currentTownName`    | 当前实时位置所在乡镇名称   |
| `taskTownCode`       | 本次任务所属乡镇代码     |
| `taskTownName`       | 本次任务所属乡镇名称     |

当前乡镇建议通过 GIS 空间匹配自动计算：

```text
无人机实时坐标点 + 乡镇行政边界面 = 当前所在乡镇
```

县区平台展示建议：

| 平台编号 | 显示名称 | 登记乡镇 | 当前乡镇 | 主载荷 | 状态 |
|---|---|---|---|---|---|
| `HD-UAV-130402-MR-0001` | 河沙镇-M350-热成像-01 | 河沙镇 | 北张庄镇 | `IR` | 飞行中 |
| `HD-UAV-130402-MR-0002` | 代召乡-Mavic3E-可见光-01 | 代召乡 | 代召乡 | `EO` | 待命 |

## 8. 外部监管 ID 映射

| 字段                           | 含义               |
| ---------------------------- | ---------------- |
| `uniqueProductId`            | 民用无人驾驶航空器唯一产品识别码 |
| `realNameRegistrationMark`   | 民航实名登记标志         |
| `remoteId`                   | 远程识别码            |
| `remoteIdModuleSerialNumber` | 运行识别模块序列号        |
| `flightServiceUavId`         | 低空飞服平台设备 ID      |
| `policeRecordNo`             | 公安备案号，如有         |
| `manufacturerSerialNumber`   | 厂商序列号            |

映射原则：

```text
平台编号 ≠ 民航实名登记号
平台编号 ≠ 唯一产品识别码
平台编号 ≠ 远程识别码
平台编号 ≠ 厂商序列号
平台编号 ≠ 公安备案号
平台编号 ≠ 飞服平台设备 ID
```

## 10. 编号生成流程

1. 选择登记归属区县。
2. 选择飞行器类别。
3. 系统读取该区县、该飞行器类别下的最大流水号。
4. 自动生成平台主编号。
5. 补充显示名称、乡镇、载荷、外部 ID 等档案字段。
6. 管理员审核确认。
7. 编号锁定。

示例：

```text
登记区县：邯山区 130402
飞行器类别：多旋翼 MR
系统生成：HD-UAV-130402-MR-0001
```

## 11. 变更规则

原则上不变更平台主编号。所属单位、运营单位、责任人、常驻乡镇、当前所在乡镇、任务用途、临时载荷、外部监管 ID 等变化，均通过字段和历史记录体现。

建议维护：

- `assetTransferHistory`：资产调拨历史
- `orgChangeHistory`：单位变更历史
- `payloadChangeHistory`：载荷变更历史
- `baseChangeHistory`：常驻点变更历史
- `externalIdBindHistory`：外部 ID 绑定历史

仅在登记错误且尚未投入使用、重复建档、设备实际不存在、数据迁移中原始档案不可追溯等情况下，考虑作废并重新建档。

## 12. 制度表述建议

> 邯郸市低空监管平台无人机设备实行“一机一档、一机一码、多码映射、全程可追溯”的编码管理制度。平台编号作为邯郸市低空监管平台内部设备主标识，由系统统一生成，采用“城市标识 + 设备类型 + 登记归属行政区划代码 + 飞行器类别代码 + 流水号”的结构。
>
> 平台编号不包含当前所属单位、运营单位、责任人、具体任务用途、乡镇街道和临时载荷信息。所属单位、运营单位、责任单位、乡镇管理、载荷能力、飞行用途等信息通过设备档案、任务记录和变更记录维护。
>
> 对接公安、民航实名登记、远程识别和低空飞行服务平台时，应分别维护唯一产品识别码、实名登记标志、远程识别码、运行识别模块序列号、飞服平台设备 ID、公安备案号等外部标识，并通过设备档案建立映射关系，不得相互替代。
