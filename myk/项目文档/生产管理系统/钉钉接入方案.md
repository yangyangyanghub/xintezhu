# 生产管理系统钉钉接入方案

## 一、项目背景

- 后端仓库：`E:\code\production-management-server`
- 前端仓库：`E:\code\SCXMGL-QDCK`
- 后端技术栈：若依 3.9.1 + Spring Boot 4.0.3 + JDK 17 + Spring Security + JWT + Redis + Flowable 7.1.0
- 前端技术栈：若依标准 RuoYi-Vue（Vue 2.6 + Element UI 2.15 + Vuex + axios + jsencrypt）
- 数据库：MySQL（带 sys_user / sys_dept 等若依标准表）

## 二、接入范围说明

钉钉接入共有 5 类典型场景，复杂度差异很大：

| 编号 | 接入类型 | 典型场景 | 复杂度 | 是否需要企业认证 |
| ---- | -------- | -------- | ------ | ---------------- |
| A | 钉钉扫码登录（SSO） | 用户用钉钉账号登录系统 | 低 | 是 |
| B | 组织架构同步 | 用户/部门数据从钉钉拉取 | 中 | 是 |
| C | 消息通知推送 | 告警、审批、待办推送钉钉 | 极低～中 | 看方式 |
| D | 钉钉审批集成 | 系统流程在钉钉发起审批 | 高 | 是 |
| E | 钉钉工作台 H5 应用 | 系统嵌入钉钉客户端 | 高 | 是 |

**推荐组合**：A（扫码登录） + B（组织同步） + C（消息推送），三者共用同一套钉钉 SDK 和 access_token 体系。

## 三、技术选型

| 组件 | 选型 | 理由 |
| ---- | ---- | ---- |
| 钉钉 SDK | 官方 dingtalk-service-sdk（新版 OpenAPI 2.0） | 阿里官方维护，JDK17 兼容好，新功能优先支持 |
| Token 缓存 | Redis（项目已有） | 钉钉 access_token 有效期 2 小时，必须缓存 |
| 模块组织 | 新建 `ruoyi-dingtalk` Maven 子模块 | 符合若依模块化风格，可独立移除 |
| 前端接入 | 登录页加扫码按钮 + 回调路由页 | 不动 axios 拦截器和 Vuex 核心结构 |

**为什么不选其他方案**：

- 老版 taobao-sdk（OpenAPI 1.0）：钉钉新功能不再补充，JDK17 有反射警告，不推荐
- 第三方封装（如 ding-talk-spring-boot-starter）：版本滞后，依赖不可控，生产不建议
- 自己 HTTP 调 OpenAPI：维护成本高，仅简单场景可用

## 四、整体架构

```
钉钉开放平台 OpenAPI
        ↑ HTTPS
        |
ruoyi-dingtalk（新增模块）
  ├─ config/    DingTalkProperties（appKey/Secret/agentId）
  ├─ client/    DingTalkClient（封装 access_token + Redis 缓存）
  ├─ service/
  │   ├─ DingAuthService     扫码登录回调
  │   ├─ DingUserSyncService 组织架构同步
  │   └─ DingMessageService  消息推送（群机器人 + 工作通知）
  └─ controller/ DingTalkController
        ↑
        |
ruoyi-framework（扩展）
  ├─ SysLoginService 新增 dingLogin() 方法
  ├─ sys_user 表    新增 dingtalk_userid 字段
  ├─ sys_dept 表    新增 dingtalk_dept_id 字段
  └─ SecurityConfig 放行 /dingtalk/** 回调路径
```

## 五、实施路线（分三期，可独立上线）

### 第一期：基础设施 + 消息通知（1～2 天）

- 申请钉钉企业内部应用，拿到 appKey / appSecret / agentId
- 新建 `ruoyi-dingtalk` 模块，封装 DingTalkClient 和 token 管理
- 实现群机器人推送 + 工作通知推送
- 在现有告警、审批节点接入推送

ROI 最高，最快出效果。

### 第二期：钉钉扫码登录 SSO（2～3 天）

- `sys_user` 加 `dingtalk_userid` 字段 + 唯一索引
- 后台用户管理页增加"绑定钉钉"功能
- 前端登录页增加"钉钉登录"按钮
- 后端回调接口 + JWT 签发对接
- 首次扫码引导绑定流程

### 第三期：组织架构同步（3～5 天）

- Quartz 定时任务（每日全量 + 启动时增量）
- 部门树映射策略（钉钉树 vs 若依树的合并规则需先和业务确认）
- 离职 / 调岗 / 转部门处理逻辑
- 字段映射：sys_user.dingtalk_userid、sys_dept.dingtalk_dept_id

### 进阶（按需）

- 第四期：钉钉审批中心对接（结合现有 Flowable 工作流引擎，双向同步审批状态）
- 第五期：钉钉工作台 H5 微应用嵌入

### 6.2 前端改造（SCXMGL-QDCK）

| 文件 | 改造内容 |
| ---- | -------- |
| `src/views/login.vue` | 在密码登录表单下方加"钉钉扫码登录"按钮，引入钉钉 DTFrameLogin 渲染二维码 |
| `src/router/index.js` | 新增 `/dingtalk/callback` 路由 |
| `src/views/dingtalk/callback.vue` | 新增回调页：拿 authCode → 调后端接口 → 拿 token → 走 `store.dispatch('Login')` |
| `src/api/login.js` | 新增 `dingLogin(authCode)` 接口方法 |

**几乎不动的部分**：

- `utils/request.js`（axios 拦截器）：token 走同一套，**不动**
- `store/modules/user.js`：复用 `Login` action，**几乎不动**
- `permission.js`（路由守卫）：**不动**
- 业务页面：**不动**

> 这就是用若依标准前端的好处：钉钉登录只是多了一种"获取 token 的方式"，token 拿到之后流程和账号密码登录完全一样。

## 七、关键技术细节

### 7.1 access_token 管理（最容易踩坑）

钉钉企业内部应用 access_token 有效期 2 小时，频繁调用会限流。

**必须用 Redis 缓存**，过期前 5 分钟自动刷新。建议封装为单例 `DingTalkTokenManager`：

```java
public String getAccessToken() {
    String cached = redisCache.getCacheObject("dingtalk:access_token");
    if (cached != null) return cached;

    String token = dingApi.getAccessToken(appKey, appSecret);
    // 实际有效期 7200 秒，缓存 7000 秒提前续期
    redisCache.setCacheObject("dingtalk:access_token", token, 7000, TimeUnit.SECONDS);
    return token;
}
```

**坑**：高并发下多个线程同时刷新 token 会导致旧 token 失效，需要加分布式锁（可用 Redisson 或 Redis SETNX 实现）。

### 7.2 扫码登录闭环流程

```
前端: DTFrameLogin → 用户扫码 → 回调 URL 带 authCode
  ↓
前端: 拿到 authCode → POST /dingtalk/login {authCode}
  ↓
后端: authCode → 调钉钉 API 换 userid
  ↓
后端: 查 sys_user WHERE dingtalk_userid = ?
  ├─ 命中 → tokenService.createToken(loginUser) → 返回 JWT
  └─ 未命中 → 返回"未绑定"错误，前端引导跳到绑定页
  ↓
前端: 收到 token → setToken(token) → store.dispatch('GetInfo') → 跳首页
```

### 7.3 老用户绑定流程

两种策略二选一：

- **策略 A**：管理员后台手动绑定（在用户管理页加"绑定钉钉"按钮）
- **策略 B**：用户首次扫码后，输入账号密码完成自动绑定

**推荐策略 B**，用户体验好，自助完成。

### 7.4 消息推送方式选择

| 方式 | 用法 | 适合场景 | 实现成本 |
| ---- | ---- | -------- | -------- |
| 群机器人（Webhook） | 直接 POST 到 webhook URL | 监控告警、构建结果、全员通知 | 极低 |
| 工作通知 API | 走 OpenAPI，需要 access_token | 个人待办、审批提醒、定向通知 | 中 |

**生产管理系统建议两个都做**：
- 异常告警 → 群机器人推到运维群
- 任务派单 → 工作通知推给具体工人

### 7.5 前端扫码登录集成方式

钉钉提供官方 JS SDK，支持 iframe 内嵌二维码模式（`DTFrameLogin`），不用跳出当前页：

```html
<!-- 在 login.vue 中引入 -->
<script src="https://g.alicdn.com/dingding/h5-dingtalk-login/0.21.0/ddlogin.js"></script>
```

```js
// 使用 DTFrameLogin 渲染二维码到指定容器
window.DTFrameLogin(
  {
    id: 'ding-login-container',
    width: 300,
    height: 300,
  },
  {
    redirect_uri: encodeURIComponent('https://your-domain.com/dingtalk/callback'),
    client_id: '你的 appKey',
    scope: 'openid',
    response_type: 'code',
    state: '随机防 CSRF 字符串',
    prompt: 'consent',
  },
  (loginResult) => {
    const { redirectUrl, authCode, state } = loginResult;
    // 拿到 authCode，调后端接口
  },
  (errorMsg) => {
    console.error('钉钉登录失败', errorMsg);
  }
);
```

## 八、避坑清单

1. **企业认证**：扫码登录、组织同步必须企业认证（要钱，1 次性几百块），群机器人不需要。可先做不需要认证的部分。
2. **appKey/Secret 千万别进 Git**：放 Nacos / 配置中心 / `application-prod.yml`（生产单独维护，加到 .gitignore）。
3. **access_token 不要每次请求都拿**：必须缓存，否则一天就超限。
4. **回调接口要做签名校验**：钉钉事件订阅回调有 HMAC 签名，不验证会被伪造。
5. **userid 不是手机号**：钉钉的 userid 是独立 ID，不要用手机号关联（用户改号就完蛋）。
6. **若依 3.9.1 + SpringBoot 4.0.3 比较新**：注意官方 SDK 是否兼容，可能需要排除某些冲突依赖（如 fastjson 版本）。
7. **JDK 17**：老版 taobao-sdk 部分依赖在 JDK17 下有反射警告，再次说明用新版 SDK。
8. **公网回调**：钉钉服务器要能访问回调地址，生产环境必须公网可达，开发可用内网穿透（如 ngrok、frp）。

## 九、风险与成本

| 项目 | 评估 |
| ---- | ---- |
| 企业认证 | 扫码登录、组织同步必须，1 次性几百块 |
| 公网回调 | 生产环境必须公网可达 |
| 总工作量 | 一期 1～2 天 + 二期 2～3 天 + 三期 3～5 天 = **6～10 人天** |
| 维护成本 | 低，钉钉 API 稳定，SDK 自动更新 |
| 不可逆改动 | 仅 sys_user / sys_dept 加字段，可向后兼容 |

## 十、待洋哥拍板的事项

1. **企业是否已经做了钉钉认证？**
   - 没认证 → 先做群机器人（不需要认证），后面升级
   - 已认证 → 直接走完整方案
2. **三期里先做哪个？** 建议按 1 → 2 → 3 顺序
3. **是否要现在开始动手第一期的代码骨架？**
   - 可以建好 `ruoyi-dingtalk` 模块、写好 DingTalkClient 和群机器人推送
   - 配置改完就能直接跑
