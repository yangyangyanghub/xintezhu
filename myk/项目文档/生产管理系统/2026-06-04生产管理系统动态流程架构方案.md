# 生产管理系统动态流程架构方案

> 文档日期：2026年6月4日  
> 前端仓库：`E:\code\SCXMGL-QDCK`  
> 后端仓库：`E:\code\production-management-server`

---

## 一、结论

结合当前前后端代码现状，推荐采用“双流分离、双引擎协同”的方案：

```text
审批工作流：Flowable + bpmn-js
项目进展流程 / 项目作业流：AntV X6
动态流程配置：pm_flow_definition + pm_flow_node_config
项目节点运行轨迹：pm_flow_node_log
业务审批关联：pm_business_process
```

当前不建议引入 LogicFlow。原因不是 LogicFlow 不适合业务流程，而是当前系统已经有 `Flowable`、`bpmn-js` 和 `@antv/x6`。继续引入 LogicFlow 会形成三套画布/流程技术并存，增加维护成本。

---

## 二、为什么这样做

### 2.1 审批工作流已经有成熟基础

后端已经接入 Flowable，并存在新版 `/workflow/*` 和旧版 `/flowable/*` 两套路由能力。相关文件包括：

- `WfModelController.java`：流程模型管理
- `WfProcessController.java`：流程运行、待办、已办、审批操作
- `WfModelServiceImpl.java`：BPMN XML 保存、模型版本、部署
- `WfProcessServiceImpl.java`：任务处理、流程详情、轨迹数据
- `FlowableConfig.java`：Flowable 引擎配置
- `project_approval.bpmn20.xml`：项目审批流程
- `contract_approval.bpmn20.xml`：合同审批流程

这说明“审批流”已经应由 Flowable 承担，不应再用 X6 或 LogicFlow 自研审批引擎。

### 2.2 前端已经有 BPMN 设计器

前端已有：

- `src/views/system/flowable/designer/index.vue`
- `src/api/flowable/process.js`
- `src/api/flowable/task.js`
- `src/api/flowable/instance.js`
- `src/api/flowable/history.js`

这条链路适合继续用于 BPMN 建模、部署、任务审批和流程轨迹展示。

### 2.3 项目进展流程不是纯审批流

生产管理系统的项目进展流程通常包含：项目信息、合同会审、任务分配、进度更新、质量审核、成果登记、成果归档、开票申请、收款登记等。

其中一部分节点是审批，一部分只是业务操作或状态记录。因此不应该把整个项目生命周期强行塞进 BPMN。

正确边界是：

```text
项目进展流程 = 项目生命周期 / 业务作业流
审批工作流 = 某个业务节点内部的审批流程
```

项目节点可以关联审批流程，但项目流程本身不等于审批流程。

### 2.4 X6 更适合项目流程图展示和布局

前端已安装 `@antv/x6`，并已有项目总流程页面：

- `src/views/project/workflow/index.vue`

当前该页面使用 CSS + SVG 手写节点和连线，后续动态节点、拖拽布局、缩放平移、状态渲染会变复杂。X6 更适合承担项目作业流、节点图、DAG、状态流程图和布局配置。

### 2.5 不建议引入 LogicFlow

当前系统已存在：

```text
bpmn-js：审批 BPMN 设计器
AntV X6：已安装，适合项目流程图
Flowable：后端审批执行引擎
```

如果再引入 LogicFlow，会出现三套图技术：

```text
bpmn-js
AntV X6
LogicFlow
```

后续会带来三套节点模型、三套数据结构、三套样式体系和三套交互事件，收益不足以覆盖复杂度。

## 三、推荐架构

```text
pm_project_info
  项目主表
  ↓
pm_flow_definition
  项目流程模板
  ↓
pm_flow_node_config
  项目流程节点配置
  ↓
pm_flow_node_log
  项目节点运行轨迹
  ↓
pm_business_process
  业务单据与 Flowable 流程实例关联
  ↓
ACT_* Flowable 表
  审批执行数据
```

职责边界：

| 模块                  | 职责                       |
| ------------------- | ------------------------ |
| Flowable            | 审批执行、任务流转、审批历史           |
| bpmn-js             | BPMN 流程建模和设计             |
| AntV X6             | 项目总流程展示、流程布局、节点状态图       |
| pm_flow_definition  | 定义项目流程模板                 |
| pm_flow_node_config | 定义节点、前置关系、坐标、跳转 URL、审批关联 |
| pm_flow_node_log    | 记录项目节点运行状态               |
| pm_business_process | 关联业务单据和 Flowable 流程实例    |

## 五、当前缺口

当前最大问题不是选型，而是“前端、文档、SQL 已经规划，后端 Java 实现没有完全补齐”。

需要重点补齐：

1. `/pm/projectInfo/operationFlow/{projectId}` 后端接口。
2. `/pm/flowDefinition/*` 后端 Controller / Service / Mapper / Entity。
3. `/pm/flowNodeConfig/*` 后端 Controller / Service / Mapper / Entity。
4. `pm_flow_node_log` 的写入时机和状态回写机制。
5. 审批完成后，Flowable 状态如何同步到项目节点状态。

---

## 六、推荐实施路径

### 阶段一：补齐后端动态流程配置接口

优先补齐：

```text
/pm/flowDefinition/*
/pm/flowNodeConfig/*
```

目标是让现有流程定义、节点配置、流程布局页面不再依赖 mock。

### 阶段二：补齐项目总流程接口

实现：

```text
GET /pm/projectInfo/operationFlow/{projectId}
```

返回项目流程节点和连线，供前端 X6 渲染。

### 阶段三：用 X6 改造项目总流程页面

改造：

```text
src/views/project/workflow/index.vue
```

建议拆分为：

```text
src/views/project/workflow/
├─ index.vue
└─ components/
   ├─ ProjectFlowGraph.vue
   ├─ ProjectFlowNode.vue
   └─ ProjectFlowLegend.vue
```

### 阶段四：用 X6 改造流程布局页

改造：

```text
src/views/flow/flowLayout/index.vue
```

该页面作为配置态，用于管理员拖拽节点、保存坐标、查看前置关系。

### 阶段五：审批中心接新版 workflow API

当前 `src/views/project/approval-pending/index.vue` 偏 mock，建议接新版：

```text
/workflow/task/todo
/workflow/task/done
/workflow/process/my
```

### 阶段六：打通业务节点和 Flowable

典型闭环：

```text
用户提交合同会审
  ↓
创建业务单据
  ↓
启动 Flowable 流程 contract_approval
  ↓
写入 pm_business_process
  ↓
写入 pm_flow_node_log，节点状态 in_progress
  ↓
审批完成
  ↓
pm_business_process = approved
  ↓
pm_flow_node_log = completed
  ↓
项目总流程图节点变绿
```

## 七、风险和注意事项

1. 不要把项目生命周期等同于审批流程。项目节点有些是审批，有些只是业务操作。
2. 不要让 X6 承担流程执行。X6 只负责图展示和交互。
3. 不建议引入第三套画布技术 LogicFlow。
4. 模板和实例要分离，运行中的项目不要被流程模板修改直接影响。
5. 审批完成后的状态回写要设计清楚，这是动态流程闭环的关键。

---

## 八、最终建议

当前系统推荐路线：

```text
审批工作流：Flowable + bpmn-js
项目进展流程：AntV X6
动态流程配置：pm_flow_definition + pm_flow_node_config
节点运行轨迹：pm_flow_node_log
业务审批关联：pm_business_process
```

这条路线能最大化复用现有代码和技术栈，避免重复建设，也能为后续动态流程、项目进展可视化、审批状态回写打下清晰边界。
