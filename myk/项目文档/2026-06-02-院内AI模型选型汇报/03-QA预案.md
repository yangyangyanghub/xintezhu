## 🔴 高概率技术问题（必须背熟）

### Q1：百炼缓存机制具体怎么实现的？

**A**：
> 「百炼用的是**显式缓存（explicit caching）**机制。客户端在请求 messages 数组里给某个 message 加 `cache_control: { "type": "ephemeral" }` 标记，百炼后端会把这部分内容做哈希缓存，TTL 5 分钟。
>
> 5 分钟内**完全相同的前缀**再次请求，命中部分按 10% 计费。
>
> 对咱们这种'同一个 system prompt + RAG 检索结果'的场景，命中率天然就高。」

**追问应对**：
- "TTL 5 分钟够吗？" → "RAG 场景的批量任务一般在 5 分钟内集中发，够用。如果不够，百炼也支持持久化缓存，单独议价"
- "需要客户端改还是服务端改？" → "都行。最优解是在 new-api 中间件统一注入，**业务前端零感知**"

---

### Q2：99% 命中率是怎么测的？真实业务能到吗？

**A**：
> 「我写了 `verify_cache_big.py`，构造了一个 12K tokens 的长 system prompt + 不同 user query，连续请求 1907 次。
>
> - 第 1 次（冷启动）：命中 0
> - 第 2 次开始：**命中 1889/1907 = 99%**
>
> 这是**理想场景**的上限。真实业务里，因为 system prompt 不完全一致、RAG 检索结果会变，**预估真实命中率 60-90%**。
>
> 这就是我**坚持先灰度 1 周再推广**的原因——拿真实数据说话，不拍脑袋。」

**追问应对**：
- "60% 也值得做吗？" → "按 60% 算，每月也能省 ¥1,100+，**改一行代码的事，ROI 是无穷大**"

---

### Q3：为什么不直接切 DeepSeek 一步到位？

**A**：
> 「三个原因：
>
> **第一**，方案 0 是**零风险**的——不换厂商、不改业务、不改模型，30 秒可回滚。方案 B 切 DeepSeek 涉及业务方适配，**至少 1-2 周**。
>
> **第二**，**两个方案不冲突**——加 `cache_control` 标记之后再切 DeepSeek，DeepSeek 也支持缓存，标记继续生效。
>
> **第三**，方案 0 的 ROI 数据可以**反过来给方案 B 的决策提供依据**——如果方案 0 跑下来月省 ¥1,800 已经够用，方案 B 可以缓做。」

---

### Q4：new-api 中间件怎么改？影响哪些调用方？

**A**：
> 「改造点很小——在 new-api 的请求转发层，**对 qwen3.6-plus / qwen3-coder-next 这两个模型**，自动在 messages 数组的 system 段尾部注入 `cache_control` 字段。
>
> 影响范围：**所有走 new-api 中转的调用方都受益，但业务代码零感知**。
>
> 改造工作量：**预计运维半天，加测试 1 天**。
>
> 我建议**先在 new-api 的灰度环境验证**，验证通过再推生产。」
