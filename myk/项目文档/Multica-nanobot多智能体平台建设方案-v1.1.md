## 一、项目背景与目标

### 1.1 背景

随着大模型技术的成熟，AI Agent 已从概念走向落地。院里各部门存在大量重复劳动场景（文档处理、数据质检、代码生成、报表生成等），但目前存在以下痛点：

| 痛点 | 说明 |
|------|------|
| **能力分散** | 各部门各玩各的，好的工具无法复用 |
| **数据安全** | 员工私用外网大模型，存在数据泄露风险 |
| **效果不可控** | 输出质量参差不齐，没有统一标准和审核机制 |
| **审计追溯难** | 谁、什么时候、用 AI 干了什么，无法追溯 |
| **投入产出不清晰** | 买了大模型账号，但不知道产生了多少价值 |

### 1.2 建设目标

**短期目标（1-2 个月）**：
- 搭建院内统一的多 Agent 协作平台
- 跑通 3-5 个高频场景（文档处理、数据质检、代码生成）
- 覆盖 3-5 个种子部门，20-30 个活跃用户
- 建立基础的安全审计机制

**中期目标（3-6 个月）**：
- 覆盖主要业务部门，100+ 活跃用户
- 沉淀 20+ 标准化 AI 工作流
- 建立院内 AI 能力市场，好的 Agent 可以共享复用
- 量化产出：人均效率提升 30%+

**长期目标（6-12 个月）**：
- 全院级 AI 基础设施，人人可用
- Agent 之间自动协作，复杂任务自动拆解执行
- 形成院里独有的 AI 知识库和能力沉淀

### 2.2 核心组件说明

| 组件 | 技术选型 | 部署方式 | 说明 |
|------|---------|---------|------|
| **Multica Server** | Multica 开源版 | Docker 部署 | 任务管理、调度、审计、权限 |
| **🧬 OpenSpace** | HKUDS/OpenSpace | 服务端统一部署 | **新增！技能进化 + Token 优化 + 院内技能市场** |
| **⚡ nanobot Worker** | HKUDS/nanobot | 每人本地运行 / 服务器集中部署 | 实际执行任务的 Agent |
| **院内大模型** | 火山引擎 / DeepSeek / 院内私有化 | API 接入 | 统一大模型出口，流量管控 |
| **MCP 工具服务** | 自研 / 开源 | 独立部署 | GIS 工具、数据库操作、文件系统等 |
| **消息渠道** | 钉钉 / 飞书机器人 | 院内 IM 对接 | 任务通知、审核交互 |

### 2.2.1 OpenSpace 核心能力（重点增值项）

| 能力 | 说明 | 业务价值 |
|------|------|---------|
| **Skill 自我进化** | 每次任务执行后，自动分析经验、生成可复用的标准化 Skill | ✅ 越用越聪明，第二次开始效率翻倍 |
| **Token 智能优化** | 自动上下文压缩 + 技能复用，平均减少 **46% Token 用量** | ✅ 大模型成本直接砍半 |
| **院内私有技能市场** | 部门 A 搞出的好技能，部门 B 直接用，全院共享经验 | ✅ 能力复用，避免重复造轮子 |
| **社区经验共享** | 可选择性接入全球社区技能市场，站在巨人肩膀上 | ✅ 别人踩过的坑，你直接用答案 |

### 2.3 关键设计原则

1. **数据不出院**：所有组件全部内网部署，无外网依赖
2. **权限最小化**：每个 Worker 只有自己工作目录的权限，公共 Worker 权限严格管控
3. **全链路可审计**：所有操作留痕，谁、什么时候、干了什么、结果是什么，全可查
4. **人工可干预**：任何环节都可以暂停、打回、人工介入审核
5. **渐进式落地**：不搞大跃进，先跑通小场景，再逐步推广

## 四、安全与合规设计

### 4.1 数据安全

| 风险 | 应对措施 |
|------|---------|
| **敏感数据泄露** | ✅ 所有数据内网传输，不经过第三方服务器<br>✅ 大模型调用走院内统一网关，数据脱敏后再出网<br>✅ 禁止私接外网大模型，所有流量走平台 |
| **代码/文档泄露** | ✅ Worker 只能访问自己工作目录下的文件<br>✅ 目录白名单机制，禁止越权访问<br>✅ 所有文件操作全审计 |
| **大模型 API Key 泄露** | ✅ Key 存在服务端，Worker 拿不到 Key<br>✅ 按人/按部门按量分配，超额自动告警 |

### 4.2 权限控制

`
┌─────────────────────────────────────────────┐
│  超级管理员（信息中心）                        │
│  • 服务器管理、权限分配、系统配置              │
├─────────────────────────────────────────────┤
│  部门管理员                                   │
│  • 本部门任务管理、本部门 Worker 配置          │
│  • 看不到其他部门数据                          │
├─────────────────────────────────────────────┤
│  普通员工                                     │
│  • 只能看自己的任务、用自己的 Worker           │
└─────────────────────────────────────────────┘
`

### 4.3 审计追溯

**每条任务全链路记录**：
- 谁在什么时候创建的任务
- 分配给了哪个 Worker，执行了多久
- 调用了哪些工具，读了哪些文件
- 生成了什么内容，修改了哪些文件
- 谁审核的，审核意见是什么
- 日志保留 ≥ 180 天

## 五、风险评估与应对

| 风险 | 发生概率 | 影响程度 | 应对措施 |
|------|---------|---------|---------|
| **员工接受度低，没人用** | 🟡 中 | 🔴 高 | 1. 先从痛点最痛的场景切入<br>2. 找几个愿意尝鲜的种子用户<br>3. 做好培训和贴身支持<br>4. 用实际效果说话，让受益者主动推广 |

| **Worker 多了服务端扛不住** | 🟢 低 | 🟡 中 | 1. Multica 本身很轻，初期 4C8G 撑 100 个 Worker 没问题<br>2. 真到瓶颈了加机器水平扩展就行 |

## 六、投入产出分析

| **大模型费用（OpenSpace 优化后）** | 0 | **~5.4 万元/年** | **平均减少 46% Token 用量，直接省一半！** |

### 6.2 产出估算（含 OpenSpace 增值）

| 产出项 | 保守估计 | 说明 |
|--------|---------|------|
| **效率提升** | 30%+ | 重复性劳动（文档、报表、代码）效率提升显著 |
