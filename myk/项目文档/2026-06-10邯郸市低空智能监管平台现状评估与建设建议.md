# 邯郸市低空智能监管平台现状评估与建设建议

## 1. 执行摘要

本次评估认为，邯郸市低空智能监管平台已具备低空监管基础框架，现网系统已经覆盖指挥调度、地图态势、设备管理、数据中心、智能分析、系统设置等核心模块，具备继续扩展为综合性低空智能监管平台的基础条件。

但从当前能力看，平台仍主要停留在“无人机指挥调度平台 + 数据展示平台”阶段，设备底账、实时态势、任务流程、成果资产、告警处置、AI 识别、外部监管协同等关键链路尚未形成完整闭环。

下一步建设重点不建议优先放在展示型大屏上，而应优先补齐四类基础能力：

1. **设备底账**：统一无人机、机巢、载荷、责任单位、区县乡镇归属。
2. **实时态势**：修复在线状态、实时位置、任务中状态、轨迹展示。
3. **任务闭环**：建立任务编号、执行、成果归档、复盘统计。
4. **问题闭环**：打通告警、AI 识别、问题派发、处置反馈、复核销号。

建议建设顺序为：**先底账治理，再实时监管，再任务闭环，最后做外部监管对接和 AI 问题闭环**。

## 2. 评估背景

本次评估对象为邯郸市低空智能监管平台现网系统。平台已具备指挥调度、地图态势、设备管理、数据中心、智能分析、系统设置等基础能力。

评估重点包括无人机设备标准化、县区和乡镇分级监管、机巢与无人机绑定关系、实时飞行态势、任务闭环、数据资产化、AI 识别闭环，以及与公安、民航实名登记、远程识别、低空飞服平台等外部系统的对接能力。

相关前置文档：[[2026-06-10邯郸市低空监管平台无人机编码规范]]。

## 3. 平台现状

已观察到的平台模块包括：现状总览、指挥调度、数据中心、智能分析、设备管理、系统设置。

当前平台规模：

| 类型 | 数量 |
|---|---:|
| 机巢 | 19 个 |
| 无人机 | 21 架 |
| 照片 | 194 张 |
| 视频 | 40 个 |

当前平台已具备 2D/3D 地图、图层管理、电力线、空间限制区范围缓冲、空间限制区范围类型、120m 以上建筑、测量工具等空间监管底座。

## 4. 总体判断

平台不是从零开始，已有低空监管基础底座；但目前更像“无人机指挥调度平台 + 数据展示平台”，距离完整的“低空智能监管平台”还需要补齐设备、任务、数据、告警、AI、外部监管对接等闭环能力。

一句话判断：**基础有了，但监管闭环还没有完全长出来。**

平台后续建设应围绕以下六条主线展开：

1. 设备标准化：把无人机、机巢、载荷、责任单位、区县乡镇归属管准。
2. 实时态势：把在线状态、实时位置、高度、速度、航向、轨迹看清。
3. 任务闭环：把任务计划、执行、监控、成果归档、复盘统计串起来。
4. 数据资产：把照片、视频、识别结果、任务成果变成可检索、可追溯、可复用的数据资产。
5. 问题闭环：把告警和 AI 识别结果转化为问题派发、处置、复核、销号流程。
6. 外部协同：与实名登记、远程识别、飞服平台、公安备案等外部监管系统形成统一映射。

## 5. 主要问题与影响分析

| 序号 | 主要问题 | 当前表现 | 影响 | 建议方向 |
|---:|---|---|---|---|
| 1 | 设备档案不够规范 | 设备名称混合单位、点位、型号、用途等信息 | 后续难以稳定关联任务、成果、告警、责任单位 | 建立平台主编号与显示名称分离机制 |
| 2 | 县区、乡镇、点位归属不清 | 缺少稳定的区县、乡镇、点位、责任单位字段 | 县乡分级监管、问题派发、属地统计无法闭环 | 补齐行政区划和责任归属字段，并结合乡镇边界自动匹配 |
| 3 | 载荷能力没有充分字段化 | 主载荷、多载荷、任务实际载荷未形成统一字段 | 无法按任务类型、载荷能力进行筛选、统计和调度 | 建立主载荷、多载荷、任务载荷字段体系 |
| 4 | 实时状态链路需要排查 | 观察到无人机离线、任务中、今日任务等指标之间存在异常 | 影响实时监管、应急调度和在线率统计可信度 | 统一机巢在线、无人机在线、任务中、位置有效口径 |
| 5 | 数据中心统计和列表存在不一致风险 | 指挥调度页有照片、视频统计，但数据中心部分列表暂无数据 | 成果资产难以追溯，统计口径可能不一致 | 排查筛选条件、权限范围、索引同步、任务成果关联关系 |
| 6 | 任务、告警、AI 闭环不足 | 任务名称临时化、测试化，告警和识别结果尚未形成处置闭环 | 平台容易停留在演示和展示层面，无法支撑监管业务 | 建立任务、告警、问题、处置、复核、销号闭环 |
| 7 | 外部监管 ID 缺少统一映射 | 后续需对接实名登记、远程识别、飞服平台、公安备案等系统 | 如果直接用外部 ID 替代平台主编号，会导致多系统关联失控 | 建立平台主编号与外部监管 ID 的映射关系 |

### 5.1 设备档案不够规范

当前设备名称存在明显不统一现象，例如“临漳资规局无人机”“观台”“M4TD”“大名自规局”“红庙中学M4TD”“束馆镇政府M4TD”“污水处理厂M4TD”等。

这些名称混合了单位、点位、型号、用途等信息，不适合作为长期稳定的平台主编号。建议采用稳定的平台主编号，显示名称服务于前端识别。主编号不应包含单位、用途、乡镇或临时载荷；这些信息应字段化管理。

### 5.2 县区、乡镇、点位归属不清

县区部署时，县级用户最关心的是设备属于哪个县区、常驻哪个乡镇、当前飞到哪个乡镇、由谁负责、今天执行了哪些任务、发现了哪些问题。

平台需要补齐县区、乡镇、点位、责任单位等字段，并结合乡镇边界图层进行空间匹配。

### 5.3 载荷能力没有充分字段化

建议建立 `primaryPayloadCode`、`payloadCodes`、`payloadConfigType`、`taskPayloadCodes` 等字段，分别表达主载荷、多载荷能力、载荷配置类型和单次任务实际载荷。

### 5.4 实时状态需要重点排查

现网指挥调度页观察到：无人机总数 21/2、任务中 0、今日任务 0、已办任务 19、离线 21。

其中“无人机总数 21/2”的指标含义需要进一步确认，可能代表总数/在线数，也可能是前端统计口径或状态字段映射异常。建议重点核实机巢在线、无人机在线、任务中、位置有效之间的状态链路是否一致。

### 5.5 数据中心统计和列表存在不一致风险

指挥调度页显示照片总数 194、视频总数 40，但数据中心页面观察到机巢列表、无人机列表、数据列表存在暂无数据情况。

可能原因包括筛选条件不同、权限范围不同、数据源不同、数据索引未同步、前端默认查询条件异常、任务设备成果之间缺少关联。

### 5.6 任务、告警、AI 闭环不足

当前任务名称存在临时化、测试化现象。告警已有入口，但需要形成“告警产生—确认—派发—处置—复核—关闭”的闭环。

智能分析已有算法仓库、识别分析、问题列表，但需要从“算法展示”升级为“问题发现闭环”。识别结果不能只停留在算法页面，应进入统一问题列表，并关联空间位置、责任单位、处置状态和复核结果。

### 5.7 外部监管 ID 需要统一映射

平台后续需要对接公安、民航实名登记、远程识别、低空飞服平台。建议不要用外部 ID 替代平台主编号，而是建立外部 ID 映射字段：`platformUavCode`、`uniqueProductId`、`realNameRegistrationMark`、`remoteId`、`flightServiceUavId`、`flightServicePlanId`、`policeRecordNo`、`manufacturerSerialNumber`。

## 6. 建设目标

建议将平台建设目标从“能展示无人机和任务”提升为“能支撑低空监管业务闭环”。

阶段性目标包括：

1. **管得准**：无人机、机巢、载荷、责任单位、区县乡镇归属清晰准确。
2. **看得清**：实时状态、实时位置、飞行轨迹、任务进度准确展示。
3. **查得到**：照片、视频、识别结果、任务成果可按设备、任务、区域、时间、问题类型检索。
4. **派得出**：告警和 AI 识别结果可转为问题，并派发给责任单位。
5. **闭得上**：问题处置、反馈、复核、销号形成完整闭环。
6. **接得通**：平台主编号与实名登记、远程识别、飞服平台、公安备案等外部监管 ID 能够统一映射。

## 10. 县乡监管视角建议

### 10.1 县级用户关注点

县级用户重点关注：

1. 本县有多少无人机和机巢。
2. 哪些设备在线，哪些设备离线。
3. 今日执行了哪些任务。
4. 哪些乡镇发现了问题。
5. 哪些问题已处置，哪些问题超期未处置。
6. 哪些任务、告警、AI 识别结果需要县级统筹。

### 10.2 乡镇用户关注点

乡镇用户重点关注：

1. 本乡镇范围内是否有飞行任务。
2. 本乡镇范围内是否发现违法建设、秸秆焚烧、河道采砂、矿山扰动等问题。
3. 问题由谁派发、谁处置、是否销号。
4. 历史问题是否反复出现。
5. 处置结果是否通过无人机复核。

### 10.3 平台能力建议

平台应支持按区县、乡镇、点位、责任单位进行筛选和统计，并在任务、成果、告警、问题四类业务对象上统一挂接行政区划信息。

## 11. 外部系统对接建议

| 外部系统 | 对接目的 | 关键字段 | 建议阶段 |
|---|---|---|---|
| 民航实名登记 | 校验无人机合法身份 | `realNameRegistrationMark`、`uniqueProductId` | 第四阶段 |
| 远程识别系统 | 获取飞行器实时识别信息 | `remoteId` | 第四阶段 |
| 低空飞服平台 | 飞行计划申报、审批、协同 | `flightServiceUavId`、`flightServicePlanId` | 第四阶段 |
| 公安备案系统 | 重点设备备案和监管协同 | `policeRecordNo` | 第四阶段 |
| 城管、自然资源、应急等业务系统 | 问题派发和处置闭环 | 问题编号、责任单位、处置状态 | 第五阶段 |

外部系统对接时，应坚持一个原则：**平台主编号保持稳定，外部系统 ID 作为映射字段管理**。不建议直接用外部系统 ID 替代平台主编号。

## 12. 最小可行版本建议

如果预算、时间或开发资源有限，建议先做 6 件事：

1. 设备编码和命名规范落库。
2. 现有 21 架无人机、19 个机巢台账治理。
3. 增加县区、乡镇、点位、责任单位字段。
4. 修复设备在线状态和实时位置展示。
5. 建立任务标准字段和成果归档关系。
6. 建立问题列表，把 AI 识别结果和人工发现问题统一纳入闭环。

这 6 件做完，平台会从“能看设备”变成“能管设备、能看飞行、能追问题”。

## 13. 建设风险与注意事项

1. **只做大屏、不治理台账，会导致平台长期无法支撑监管。** 大屏展示依赖底层数据，设备、任务、成果、问题关系不稳定时，展示越多，口径矛盾越明显。
2. **只接入设备、不打通任务和成果，平台只能看飞行，不能管业务。** 无人机监管平台必须回答“飞了什么、发现了什么、谁来处理、是否处理完”。
3. **只做 AI 识别、不做问题派发和销号，智能分析会停留在演示层面。** AI 识别结果必须进入问题闭环，否则难以转化为监管成效。
4. **外部系统 ID 不能替代平台主编号，否则后续多系统对接会失控。** 平台主编号应保持稳定，外部 ID 应统一映射。
5. **实时状态口径必须统一，否则在线率、任务数、离线数会互相矛盾。** 建议先统一状态模型，再做展示统计。
6. **县乡边界和责任单位不清，会影响问题派发和考核。** 属地监管场景下，空间归属和责任归属是闭环基础。

## 14. 汇报口径建议

> 现有平台已经具备低空监管平台的基础框架，包括指挥调度、地图态势、设备管理、数据中心、智能分析和系统设置等核心模块。下一步建设重点不是推倒重做，而是在现有基础上补齐“设备标准化、实时态势、任务闭环、数据资产、外部监管对接、AI 问题处置”六条主线。

简化表达：

1. 先把设备管准。
2. 再把飞行看清。
3. 最后把问题闭环。

正式表述建议：

> 不建议优先投入过多资源建设展示型大屏。平台真正的建设重点应放在设备底账、实时状态、任务流程、成果资产和问题闭环上。只有底层数据和业务闭环稳定，展示层能力才具备长期价值。

## 15. 结论

优先级建议：先治理设备台账和编码，再打通实时状态和乡镇归属，然后补任务、成果、告警、AI 闭环，最后做外部监管系统对接和县乡监管看板。

平台建设不应只追求展示效果，而应优先夯实设备底账、实时状态、任务流程、成果资产和问题闭环。展示层是平台形象，底层数据和业务闭环才是平台长期可用、可管、可扩展的基础。
